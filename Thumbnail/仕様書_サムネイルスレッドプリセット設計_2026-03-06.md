# 仕様書: サムネイルスレッドプリセット設計（2026-03-06）

## 1. 目的
- サムネイル作成の負荷プリセットを、ユーザーに説明可能な形で固定する。
- プリセット名と内部挙動の対応を明確にし、今後のUI実装と並列制御実装の基準とする。

## 2. 前提
- プリセットはユーザーが選ぶ「負荷感」の定義であり、内部実行枠数と完全一致しなくてもよい。
- ただし、プリセットの大小関係と体感負荷は直感に反しないこと。
- 通常レーン、低速レーン、リカバリーレーンの3責務は維持する。
- エラーまたは高負荷を感知した場合は、プリセット値より安全側の抑制制御を優先してよい。
- エンジン外部化後も成立するよう、高負荷検知の意思決定はエンジン外で行う前提とする。
- `CommonSettingsWindow` には既に `FileIndexProvider` 3択UIが実装済みであるため、サムネイル設定追加は共有画面の整合を前提とする。
- 管理者権限を要する経路は、`Watcher` 側の `usnmft = AdminUsnMft` と衝突しない共通基盤へ寄せる前提とする。

## 3. プリセット一覧

### 3.1 slow
- 位置付け:
  - 最低負荷優先。
- 想定用途:
  - 作業中の裏実行。
- 表示上の意味:
  - `1スレッド相当`
- 内部方針:
  - 低負荷モードとして扱う。
  - CPU/I/O抑制を優先する。
  - 内部では救済レーン飢餓を防ぐため、実行枠の厳密1固定は必須としない。
- 期待挙動:
  - バックグラウンドでゆっくり進む。

### 3.2 normal
- 位置付け:
  - 標準運用。
- 想定用途:
  - 常用。
- 表示上の意味:
  - 標準速度。
- 内部方針:
  - 通常レーンを優先しつつ、低速レーンとリカバリーレーンを必要時に確保する。
- 期待挙動:
  - 速度と安定性のバランスを取る。

### 3.3 ballence
- 位置付け:
  - 安定寄りバランス。
- 想定用途:
  - 大動画と通常動画が混在する環境。
- 表示上の意味:
  - 通常より少し安全側。
- 内部方針:
  - 通常レーンの独走を抑え、巨大動画や再試行動画の処理開始機会をやや厚めに保証する。
- 期待挙動:
  - 詰まりを減らしつつ速度も確保する。

### 3.4 fast
- 位置付け:
  - 速度重視。
- 想定用途:
  - 早く処理を終えたい通常時。
- 表示上の意味:
  - 論理コアの半分程度を使う高速モード。
- 内部方針:
  - 通常レーンに多くの枠を割り当てる。
  - ただし巨大動画と再試行動画が完全に飢餓しないよう予約条件を維持する。
  - エラーや高負荷を感知した場合は、一時的に安全側へ縮退する。
- 期待挙動:
  - 通常動画のスループットが最優先になる。

### 3.5 max
- 位置付け:
  - 上限近くまで使う最大性能モード。
- 想定用途:
  - 専用処理時、他作業が少ない時。
- 表示上の意味:
  - 安全上限まで積極利用。
- 内部方針:
  - 実行上限まで並列度を上げる。
  - 動的並列制御による減速・復帰は維持する。
  - エラーや高負荷を感知した場合は、通常より積極的に抑制してよい。
- 期待挙動:
  - 最速だが、環境依存で負荷変動が大きい。

### 3.6 custum
- 位置付け:
  - 手動設定。
- 想定用途:
  - 上級者向け調整。
- 表示上の意味:
  - 並列数・閾値・必要に応じて予約方針を個別に決める。
- 内部方針:
  - ユーザー設定値を優先する。
- 期待挙動:
  - 利用者が直接チューニングできる。

## 4. 推奨並列方針

| プリセット | 推奨並列基準 | 主眼 |
|---|---|---|
| `slow` | `1スレッド相当` | 最低負荷 |
| `normal` | 論理コアの `1/3` 前後 | 常用 |
| `ballence` | 論理コアの `1/4〜1/3` | 詰まり回避 |
| `fast` | 論理コアの `1/2` | 速度重視 |
| `max` | 安全上限まで | 最大性能 |
| `custum` | 手動指定 | 個別調整 |

## 5. レーン予約方針
- 通常レーン:
  - 通常動画の主処理を担当する。
- 低速レーン:
  - 巨大動画を担当する。
- リカバリーレーン:
  - 再試行動画と救済処理を担当する。

### 5.0 高負荷検知の責務
- エンジン:
  - ジョブ単位の実行結果と局所メトリクスを返す。
- オーケストレータ:
  - 並列度、レーン予約、縮退、復帰を決定する。
- 管理者権限サービス:
  - Disk温度、UsnMft、将来のセンサー情報など、特権が必要な取得を担当する。

### 5.0.1 管理者権限サービスの前提
- Disk温度取得は管理者権限が必要になる可能性を前提とする。
- UsnMft利用も同様に、管理者権限サービスへ寄せる前提で設計する。
- 本体アプリとエンジンは、サービスから受け取った負荷情報を参照するだけに留める。
- `Watcher` 側の `UsnMftProvider` は `AdminUsnMft` 専用へ分離済みであるため、将来の管理者権限サービスはUsnMft状態取得とDisk温度取得を共存できる構成を優先する。

### 5.0.2 IPC連携の基本方針
- 本体アプリはオーケストレータとして、エンジンと管理者権限サービスの両方から情報を受け取る。
- エンジンはジョブ単位の結果通知を行い、グローバル負荷判定は行わない。
- 管理者権限サービスは高負荷判定に必要な外部シグナルを返すが、並列度の最終決定は行わない。
- サービス未接続時は、内部メトリクスのみで `high-load` 判定を代替する。

### 5.0.3 IPC項目一覧

| 項目名 | 呼び出し元 | 提供側 | 用途 | 主な返却・通知項目 | 必須 |
|---|---|---|---|---|---|
| `GetSystemLoadSnapshot` | オーケストレータ | 管理者権限サービス | 全体負荷判定 | CPU使用率, I/O待ち傾向, メモリ圧迫, 取得時刻 | 任意 |
| `GetDiskThermalSnapshot` | オーケストレータ | 管理者権限サービス | Disk温度由来の縮退判断 | Disk識別子, 温度, 閾値超過状態, 取得時刻 | 任意 |
| `GetUsnMftStatus` | オーケストレータ | 管理者権限サービス | I/O負荷や監視基盤状態の把握 | ボリューム, 利用可否, 監視状態, エラー情報 | 任意 |
| `ReportEngineJobMetrics` | エンジン | オーケストレータ | 局所メトリクス収集 | ジョブID, 処理時間, 失敗理由, 再試行回数, デコード結果 | 必須 |
| `NotifyThrottleDecision` | オーケストレータ | UI/ログ層 | 並列抑制・復帰理由の可視化 | 適用並列数, 理由種別, 原因詳細, 適用時刻 | 必須 |
| `GetThrottlePolicySnapshot` | UI/監視 | オーケストレータ | 現在の制御状態表示 | 元プリセット, 現在並列数, 縮退理由, 復帰待ち状態 | 任意 |

### 5.0.3.1 既存FileIndexProviderとの整合
- `FileIndexProvider` の `ProviderDisplayName` と、サムネイル側のプリセット表示は別責務として扱う。
- 設定画面上では「インデックス方式」と「サムネイル負荷プリセット」を近接させても、意味を混同しない見出しに分ける。
- `usnmft` が非管理者環境で `availability_error:AdminRequired` を返した場合でも、サムネイル高負荷制御の失敗とは混同しない。

### 5.0.4 IPC DTO定義案

#### `EngineJobMetricsDto`
- `JobId`

- `SourcePath`

- `LaneKind`
  - `Normal` / `Slow` / `Recovery`。
- `ElapsedMs`
  - 実行時間。
- `Succeeded`
  - 成否。
- `FailureKind`
  - `None` / `Io` / `Decode` / `Index` / `Timeout` / `Unknown`。
- `AttemptCount`
  - 実行時点の再試行回数。
- `DecodedFrameCount`
  - デコード済みフレーム数。
- `PeakWorkingSetMb`
  - 実行中の最大使用メモリ目安。
- `CapturedAtUtc`
  - 記録時刻。

#### `SystemLoadSnapshotDto`
- `CpuUsageRate`
  - プロセスまたは全体CPU使用率。
- `IoBusyRate`
  - I/O飽和の近似指標。
- `MemoryPressureRate`
  - 使用可能メモリ圧迫の近似指標。
- `QueueBacklogCount`
  - 保留ジョブ総数。
- `SlowLaneBacklogCount`
  - 巨大動画待機数。
- `RecoveryLaneBacklogCount`
  - 再試行待機数。
- `SampleWindowMs`
  - 集計窓。
- `CapturedAtUtc`
  - 記録時刻。

#### `DiskThermalSnapshotDto`
- `DiskId`
  - 物理ディスク識別子。
- `TemperatureCelsius`
  - 温度。
- `WarningThresholdCelsius`
  - 警告閾値。
- `CriticalThresholdCelsius`
  - 危険閾値。
- `ThermalState`
  - `Normal` / `Warning` / `Critical` / `Unavailable`。
- `CapturedAtUtc`
  - 記録時刻。

#### `UsnMftStatusDto`
- `VolumeName`
  - 対象ボリューム名。
- `Available`
  - 利用可否。
- `LastScanLatencyMs`
  - 直近走査時間。
- `JournalBacklogCount`
  - 未処理量の目安。

  - `Ready` / `Busy` / `Unavailable` / `AccessDenied`。

- `DecisionKind`
  - `Keep` / `ThrottleDown` / `RecoverUp`。
- `ReasonKind`
  - `Error` / `HighLoad` / `Thermal` / `Manual` / `Fallback`。
- `ReasonDetail`

- `CapturedAtUtc`
  - 記録時刻。

### 5.0.5 high-load 判定式草案
- high-load 判定は単一指標ではなく、複数シグナルの合成スコアで扱う。
- 第1段階の内部メトリクス版では、次の5入力を採用する。
  - `ErrorScore`
  - `QueuePressureScore`
  - `SlowBacklogScore`
  - `RecoveryBacklogScore`
  - `ThroughputPenaltyScore`
- 第1段階の合成スコア例:
  - `HighLoadScore = ErrorScore * 0.30 + QueuePressureScore * 0.30 + SlowBacklogScore * 0.15 + RecoveryBacklogScore * 0.15 + ThroughputPenaltyScore * 0.10`
- 第1段階では `CpuScore` `IoScore` `MemoryScore` を採用しない。
- 理由:
  - UIメーター依存を避けるため。
  - サービス未接続でも成立させるため。
- 各スコアは `0.0` から `1.0` に正規化する。
- 各入力の一次ソースは以下とする。
  - `ErrorScore`
    - `batchFailedCount`
    - `AutogenTransientFailureCount`
    - `FallbackToFfmpegOnePassCount`
    - `AutogenRetrySuccessCount`
  - `QueuePressureScore`
    - `activeCountAfterBatch`
    - `currentParallelism`
    - `configuredParallelism`
  - `SlowBacklogScore`
    - `hasSlowDemand`
  - `RecoveryBacklogScore`
    - `hasRecoveryDemand`
  - `ThroughputPenaltyScore`
    - `batchProcessedCount`
    - `batchMs`
- 基本判定:
  - `0.00 - 0.54`
    - 通常運転。
  - `0.55 - 0.74`
    - 軽度高負荷。1段階縮退候補。
  - `0.75 - 0.89`
    - 高負荷。即時縮退。
  - `0.90 - 1.00`
    - 危険域。最小安全並列まで落とす。
- `DiskThermalSnapshotDto.ThermalState == Critical` は、合成スコアに関係なく強制縮退条件とする。
- `FailureKind == Io` または `FailureKind == Timeout` の連続発生は `ErrorScore` を強めに補正する。
- `SlowBacklogScore` と `RecoveryBacklogScore` は、第1段階では presence ベース、将来は待機数と最古待機時間へ拡張してよい。
- 復帰条件:
  - `HighLoadScore <= 0.45` が一定期間継続した時のみ1段階復帰する。
  - 段階復帰の待機時間は、直前の縮退理由が `Thermal` の時は長めに取る。
- `slow` プリセット時は `ConfiguredParallelism` 自体が低いため、縮退より待機延長を優先してよい。
- `usnmft` の可用性低下はファイルインデックス経路の問題であり、サムネイル `HighLoadScore` へ直接混ぜず、別ログ軸で扱う。
- UIメーター用 `PerformanceCounter` 値は、内部メトリクス版 `HighLoadScore` の一次入力へ使わない。

### 5.1 予約の原則
- 予約枠は「常時固定」ではなく「対象ジョブが存在する時だけ有効」を原則とする。
- 理由:
  - 空振り予約で通常スループットを落としすぎないため。

### 5.2 slow の扱い
- `slow` は低負荷優先のため、予約枠の存在保証より負荷抑制を優先する。
- ただし、再試行動画や巨大動画が永続的に詰まらないよう内部救済は残す。

## 6. UI要件
- プリセット名の選択UIを持つこと。
- `custum` 選択時のみ、個別並列数とレーン閾値の直接編集を前面に出すこと。
- `slow` は「低負荷バックグラウンド」、`fast` は「論理コア半分利用」を明記すること。
- プリセット切替時に、想定される負荷感と挙動を短文で表示できること。

## 7. ログ要件
- プリセット適用時に、適用名・解決並列数・予約有効条件をログへ残すこと。
- 実行中の並列変更がある場合も、元プリセットと動的調整結果を追跡できること。
- 並列抑制の理由が `error` か `high-load` かを区別して追跡できること。
- `high-load` の情報源が、内部メトリクスかサービス由来かを区別できること。
- IPC未接続、取得失敗、権限不足を区別して追跡できること。

## 8. 互換方針
- 既存の閾値プリセット（軽量動画重視 / バランス / 巨大動画重視）は廃止ではなく内部マッピング候補として扱う。
- 過去設定値が存在する環境では、既存の並列数・閾値を壊さず `custum` へ吸収できるようにする。
- 将来サービス未接続の環境でも、内部メトリクスだけで最低限の高負荷縮退が成立するようにする。

## 9. 今後の見直し条件
- `slow` が実際には重いと評価された場合。
- `fast` の論理コア半分利用でI/O飽和が常態化した場合。
- `ballence` が `normal` と体感差を出せない場合。
- 実運用で「常時予約」の方が有利と観測された場合。
- 高負荷感知の閾値が厳しすぎて、常に縮退する状態になった場合。
