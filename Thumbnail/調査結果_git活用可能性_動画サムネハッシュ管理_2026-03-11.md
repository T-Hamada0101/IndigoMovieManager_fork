# 調査結果（git活用可能性: 動画・サムネイルのハッシュ管理）

最終更新日: 2026-03-11

## 1. 結論

- `git` を動画・サムネイルの実運用管理レイヤーへ入れても、現行課題の高速化には効きにくい。
- 特に `git` を「動画本体の管理」「サムネイル保存庫」「変更検知の主経路」に使う案は非推奨。
- 理由は単純で、現行の主要ボトルネックが `git` の得意分野ではないため。
  - 監視フォルダの列挙は既に `Everything` / `USN-MFT` / Filesystem フォールバックで高速化済み。
  - 重い区間は新規動画の `MovieInfo` 生成、DB登録、UI反映側に寄っている。
- 例外的に、監査用の manifest や差分レポートをテキストで `git` 管理する用途なら有効。
  - ただしこれは「履歴管理」には効くが、「実行時高速化」にはほぼ効かない。

## 2. 現状整理

### 2.1 ハッシュの役割

- `DB/SQLite.cs`
  - `movie` テーブルに `hash` 列を持つ。
- `Thumbnail/Tools.cs`
  - `GetHashCRC32` は動画ファイルの先頭 `128KB` だけを読み、`CRC32` を計算する。
  - 目的は「完全同一性証明」ではなく、軽量に安定した識別子を作ること。
- `Models/MovieInfo.cs`
  - 新規動画の DB 登録時に `Hash = Tools.GetHashCRC32(rawPath)` を実行する。
- `Thumbnail/ThumbnailMovieMetaCache.cs`
  - サムネイル作成時は `hashHint` があればそれを優先し、無い場合だけ `GetHashCRC32` を再実行する。
- `Thumbnail/ThumbnailPathResolver.cs`
  - サムネイルの命名規則は `"{body}.#{hash}.jpg"`。
  - つまり hash はサムネイルファイル名の安定化にも使われている。

### 2.2 監視・列挙の現状

- `Watcher/MainWindow.Watcher.cs`
  - MainDB の `movie_path` と `hash` を辞書化して存在確認している。
- `Watcher/EverythingFolderSyncService.cs`
  - 監視フォルダの列挙は `Everything IPC` を優先し、高速経路を持っている。
- `src/IndigoMovieManager.FileIndex.UsnMft/*`
  - `USN-MFT` ベースのファイルインデックス経路も存在する。
- `Thumbnail/Implementation Plan_監視フォルダ追加スキャンDB登録高速化.md`
  - ローカル NTFS は `Everything`、非 NTFS/NAS は既存監視へフォールバックする方針が既に採用されている。

### 2.3 今の設計で残っているムダ

- `src/IndigoMovieManager.Thumbnail.Queue/QueuePipeline/QueueRequest.cs`
  - Queue 永続化時に `Hash` を持っていない。
- `src/IndigoMovieManager.Thumbnail.Queue/QueueDb/QueueDbService.cs`
  - QueueDB も `Hash` を保持していない。
- `Thumbnail/MainWindow.ThumbnailCreation.cs`
  - そのため QueueDB リース経路では `MovieId/Hash` が欠落し得て、UI一覧や MainDB から補完している。
- これは `git` 導入より先に潰すべき、実装上の素直な改善余地。

## 3. git案ごとの評価

| 案 | 判定 | 理由 |
|---|---|---|
| 動画本体を `git` で管理する | 非推奨 | `git` は blob 化のために内容全体を読む前提で、巨大動画では I/O コストが重い。履歴肥大化、GC、排他、LFS 運用も重い。 |
| サムネイルを `git` で管理する | 非推奨 | 生成 JPEG は churn が激しく、削除しても履歴が残る。アプリは平文の実ファイル存在確認を前提にしており、相性が悪い。 |
| 変更検知を `git status` / `git diff` で行う | 非推奨 | `git` は repo 配下かつ追跡対象前提。未 commit / 未 add / repo 外の動画群には弱く、監視用途では `Everything` / `USN-MFT` の方が自然。 |
| 現行 CRC32 の代わりに `git` ハッシュを使う | 非推奨 | 現行 hash は「先頭128KBだけ読む軽量キー」。`git` の内容アドレス hash は用途が重く、互換性も壊す。 |
| manifest や監査ログだけ `git` で持つ | 条件付きで可 | 実行時高速化には効かないが、履歴比較やチーム共有には使える。 |

## 4. 性能観点の整理

### 4.1 現行 hash はすでに軽い

- `Thumbnail/Hash取得ベンチ結果_2026-02-25.md`
  - `System.IO.Hashing` ベースの `CRC32` へ置き換え済み。
  - 既存互換を保ちながら旧式実装より約 `5倍` 速い。
- しかも現行は「先頭 `128KB` のみ」を読む。
  - `git` の内容ハッシュのように「ファイル全体を読む」前提とはコスト構造が違う。
  - 10GB 動画なら、`128KB` と `10GB` では I/O の桁が全く違う。

### 4.2 実測上の律速は別にある

- `Thumbnail/調査結果_監視フォルダ追加スキャンDB登録ボトルネック_2026-02-24.md`
  - `scanned=2034, new=2034` の重負荷ケースで全体 `388秒`。
  - QueueDB の反映遅延は平均 `157ms` 程度で主因ではない。
  - 主因は以下と整理されている。
    - `MovieInfo` 生成
    - DB登録
    - UI反映
- つまり、今の遅さは「git で差分を持てば解決する種類の遅さ」ではない。

### 4.3 監視用途では git より既存経路が強い

- `git` は履歴管理システムであって、OS ファイル変更通知や NTFS インデックスではない。
- このプロジェクトは既に
  - `Everything`
  - `USN-MFT`
  - Filesystem フォールバック
  という、動画ライブラリ監視に合った経路を持っている。
- 特にローカル NTFS の「大量ファイル列挙・差分取得」は `git` より既存方針の方が筋が良い。

## 5. gitより優先すべき改善

## 5.1 QueueDBに hash を保持する

- Queue に `Hash` を載せれば、Consumer 側の `MovieId/Hash` 補完を削減できる。
- 効果:
  - MainDB 再参照の削減
  - UI一覧参照の削減
  - `hashHint` 欠落による再計算の抑制

## 5.2 監視登録時の重処理を段階化する

- 大量投入時は `MovieInfo` のフル生成が支配的。
- 先に最低限だけ登録し、重いメタ情報は後追い補完する設計の方が `git` より効く。
- 既存資料でも「軽量登録 + 後追い補完」は候補として挙がっている。

## 5.3 変更判定を軽量キー優先にする

- `path + size + lastWriteUtc`、必要なら `USN/FileId` を先に使い、変化が疑われる時だけ `CRC32` を計算する。
- `ThumbnailMovieMetaCache` も `path|length|lastWriteTicks` をキーにしており、この方向と整合する。

## 5.4 履歴が欲しいなら runtime ではなく export を versioning する

- もし「いつ何が増えたか」を追いたいなら、
  - MainDB の軽量スナップショット
  - サムネイル未作成一覧
  - 失敗一覧
  を JSON / CSV へ export し、その成果物だけ `git` で管理する方が安全。
- これはアプリの実行系に `git` を埋め込まないので、設計汚染が少ない。

## 6. 推奨方針

- 不採用:
  - `git` / `Git LFS` を動画・サムネイルの runtime 管理基盤として使う案
- 採用:
  - 現行の `CRC32 + MainDB + Everything/USN-MFT + QueueDB` 方針を継続
- 優先改善:
  1. QueueDB に `Hash` を持たせる
  2. 大量スキャン時の `MovieInfo` 生成を軽量化または段階化する
  3. MainDB 登録と UI 反映のバッチ化を継続する

## 7. 次アクション案

1. QueueDB へ `Hash` 列を追加した場合の影響範囲を棚卸しする。
2. 大量投入時の「軽量登録モード」の成立条件を整理する。
3. `MovieInfo` 生成時間、DB登録時間、UI反映時間の最新実測を取り直し、改善優先度を再確定する。
