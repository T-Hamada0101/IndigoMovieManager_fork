# 要件定義書: サムネイルスレッド制御とレーン設計（2026-03-06）

## 1. 目的
- サムネイル作成のスレッド制御を、速度・安定性・再試行救済の3点で整理する。
- ユーザー設定値と内部スケジューリングの責務を分離し、今後の実装判断をぶらさない。
- 低速化が必要なジョブと、速度優先で捌きたい通常ジョブを分けて扱えるようにする。

## 2. 背景
- 通常動画を速度重視で捌きたい一方、巨大動画や失敗動画が通常処理に混ざると全体が詰まりやすい。
- 失敗動画の中には動画自体の破損だけでなく、I/O揺らぎや一時的な外部要因も含まれる。
- そのため、通常レーンとは別に、巨大動画と再試行動画を低速で安全に回す仕組みが必要である。

## 3. 用語定義
- 通常レーン:
  - 速度重視で通常動画を処理する主レーン。
- 低速レーン:
  - 巨大動画を低速で処理するレーン。
- リカバリーレーン:
  - エラー発生後の再試行動画を低速で処理するレーン。
- 再試行動画:
  - 直前までの処理でエラーとなり、再投入された動画。
- 低負荷モード:
  - ユーザーが「1スレッド相当」を選んだときの、バックグラウンド優先動作。

## 4. 基本方針
- ユーザー設定は「PCへどの程度負荷を掛けるか」を表す。
- 内部スケジューリングは「詰まりをどう防ぐか」を表す。
- そのため、ユーザー設定値と内部の実行枠数は完全一致を必須としない。
- ただし、ユーザー設定が小さいほど CPU / I/O / メモリ圧迫を抑えることを必須とする。

## 5. 機能要件

### 5.1 ユーザー設定
- ユーザーはサムネイル作成の負荷プリセットを選択できること。
- プリセット名は仮称として以下を用意すること。
  - `idle`
  - `slow`
  - `normal`
  - `ballence`
  - `fast`
  - `max`
  - `custum`
- 将来的に表記修正 (`balance`, `custom`) は可能とするが、現時点では仮称として扱う。

### 5.2 低負荷モード
- 低負荷モードは `slow` と `idle` の2段階を持てること。
- このとき優先すべき要件は、絶対的な処理速度ではなく、体感負荷の低さであること。
- 低負荷モードでは、通常操作の邪魔をしないよう CPU / I/O 負荷を抑制すること。
- 低負荷モードでも、再試行動画や巨大動画により通常動画が完全停止しないこと。

### 5.2.1 slow
- `slow` は「軽いバックグラウンド運転」として扱うこと。
- `slow` では、進捗や救済レーンを維持しつつ、通常より明確に負荷を下げること。
- `slow` のワーカー優先度候補は `BelowNormal` とする。

### 5.2.2 idle
- `idle` は「触感優先の最小干渉モード」として扱うこと。
- `idle` では、通常レーン常時1本、プレビュー停止または強い遅延、UI更新強抑制を行えること。
- `idle` のワーカー優先度候補は `Idle` とする。
- `idle` は `slow` より遅くてもよいが、触っている間の邪魔を最小化することを優先する。

### 5.3 通常モード
- 通常時は速度重視で、通常レーンを優先して回すこと。
- `fast` プリセットでは、論理コア数の半分程度を通常処理に使えること。
- `max` プリセットでは、環境上安全と判断した上限まで並列度を上げられること。

### 5.4 リカバリーレーン
- 一度エラーになった動画は、再試行時にリカバリーレーンで処理すること。
- リカバリーレーンは低速・安全重視とし、通常レーンを詰まらせないこと。
- リカバリーレーンでは、必要に応じて動画インデックス再構築などの救済処理を実行できること。
- エラー原因が動画自体の破損ではなく、I/O揺らぎ等であっても、再試行動画は同じ低速レーンで扱うこと。

### 5.5 巨大動画の低速レーン
- 極端に時間がかかる巨大動画は、通常レーンではなく低速レーンへ回せること。
- 巨大動画が通常レーンに混入しても、他の通常動画の進行を長時間止めないこと。
- 巨大動画判定の閾値は設定で調整可能とすること。

### 5.6 動的並列制御
- エラー率や失敗傾向に応じて、スレッド数を動的に調整する機能は維持すること。
- 既存の動的並列制御は撤廃せず、改善対象として扱うこと。
- エラー増加時は並列度を抑え、安定時は徐々に戻すこと。
- エラー増加時だけでなく、高負荷を感知した時も並列度を抑えられること。
- 高負荷要因が解消した時は、段階的に復帰調整できること。
- 動的制御により、通常レーン・低速レーン・リカバリーレーンのいずれかが長時間飢餓状態にならないこと。

## 6. 設計要件

### 6.1 レーン分離
- 通常レーン、低速レーン、リカバリーレーンは論理的に分離すること。
- 分離は専用スレッド固定でも、共有ワーカープールへのゲート制御でもよい。
- ただし、ユーザーから見て各レーンの責務が明確であること。

#### 6.1.1 難読動画成果の反映方針
- 通常レーンは初回 `autogen` を担当し、失敗後の重い救済は保持しないこと。
- 低速レーンは `retry policy` によって選ばれた長尺 `No frames decoded` 群、超短尺 tiny-seek 群、repair/remux 群を担当すること。
- 移送条件は動画名ではなく `decision_basis` で表現すること。
- `repair` は「全編復旧」だけでなく、「先頭有効区間だけ救出してサムネ取得する」責務を持てること。

### 6.2 高負荷検知の責務分離
- 高負荷検知の最終判断は、エンジン本体ではなく上位のオーケストレータ側で行うこと。
- エンジンは、ジョブ単位の局所メトリクス（処理時間、失敗理由、再試行回数、デコード失敗など）を返す責務に留めること。
- CPU負荷、I/O待ち、Disk温度、巨大動画滞留、リカバリ滞留などの全体負荷判断は、複数エンジン横断で扱える層に集約すること。
- 第1段階の内部メトリクスは、UIメーター値ではなく `QueueProcessor` と `ThumbnailEngineRuntimeStats` で取得できる値を一次入力とすること。

### 6.3 管理者権限サービス前提
- Disk温度取得やUsnMft利用など、管理者権限を要する機能は本体UIやエンジンへ直接埋め込まないこと。
- これらの取得は、将来的に管理者権限サービスへ分離できる構成を前提とすること。
- サービスは特権が必要な情報取得を担当し、本体側はその結果を受けて並列度調整を行うこと。
- 既に `Watcher` 側で `usnmft = AdminUsnMft` の特権境界が明確化されたため、サムネイル側の特権機能も別サービスを乱立させず、共通の管理者権限サービス基盤へ寄せること。
- 管理者権限サービスは、Disk温度取得、UsnMft状態取得、サービス能力開示までを責務とし、縮退・復帰・レーン予約の最終判断は持たないこと。
- `Watcher` の `AdminUsnMft` とサムネイル側の Disk温度取得は、同じ `AdminTelemetry` サービスホスト配下のモジュールとして扱い、別pipe・別サービスを作らないこと。

### 6.4 IPC連携前提
- 本体UI、オーケストレータ、外部化エンジン、管理者権限サービスは、将来的にIPCで疎結合に連携できること。
- IPC未接続時でも、内部メトリクスのみで最低限の縮退制御と復帰制御が成立すること。
- IPCは「負荷スナップショット取得」「エンジン実行結果通知」「縮退判断反映」を最低限の責務として持つこと。
- 管理者権限サービスが停止または未導入の場合でも、サムネイル作成自体は継続できること。

### 6.5 予約枠
- リカバリーレーンは、通常負荷時でも処理開始機会を失いにくいこと。
- 巨大動画用の低速レーンも、通常動画に埋もれず処理開始できること。
- ただし、予約枠が空振りする場合は、全体スループットを不必要に落とさないこと。

### 6.6 低負荷優先
- ユーザーが低負荷モードを選択した場合、内部実行枠の定義よりも、実際の負荷抑制を優先すること。
- 低負荷モードでは、スレッド優先度、待機時間、I/O間隔などを含めて総合的に負荷を下げること。

### 6.7 共有設定UI整合
- `CommonSettingsWindow` はファイルインデックス3プロバイダ設定の反映先でもあるため、サムネイル設定追加で既存の `FileIndexProvider` UIを壊さないこと。
- 設定画面の追記は、プロバイダ選択、サムネイルプリセット、既存レーン閾値が同居しても理解しやすい配置とすること。
- 共通設定画面に変更を入れる際は、Watcher側完了タスクとの回帰を前提に扱うこと。

### 6.8 進捗タブ即時切替
- サムネイル進捗タブからも、現在の負荷プリセットをドロップダウンで変更できること。
- 進捗タブでの変更は、設定画面の値と二重管理にせず、同一設定値へ即時保存されること。
- 進捗タブでプリセットを変更した場合、進捗表示上のラベル、解決並列数表示、実行制御が同一セッション内で追従すること。
- 動的縮退中であっても、進捗タブ上で「選択モード」と「現在の実効状態」を見分けられること。

## 7. 非機能要件
- 通常レーンのスループットを、巨大動画や再試行動画が継続的に阻害しないこと。
- 再試行動画の救済処理により、通常ジョブの平均待機時間が過度に悪化しないこと。
- 高負荷状態を感知した際に、アプリ全体の体感劣化を抑える方向へ制御できること。
- 第1段階の高負荷検知は、UI未起動またはUIメーター無効でも成立すること。
- 並列度変更やレーン振り分け理由をログで追跡できること。
- プリセット変更時に、現在の構成と意図がUI上で理解しやすいこと。

## 8. 受け入れ基準
- `1スレッド相当` 選択時に、バックグラウンドで低負荷動作すること。
- 通常モードでは、通常動画のスループットを優先できること。
- 再試行動画が通常レーンではなく低速側で処理されること。
- 巨大動画が通常レーンを長時間占有しないこと。
- `fast` で論理コア半分程度を使う構成を選べること。
- エラー発生時の動的並列調整が維持されること。
- `idle / slow / normal / ballence / fast / max / custum` の各プリセット概念を保持できること。

## 9. 実装判断時の注意
- 「ユーザー設定1 = 内部スレッド厳密に1本」を必須にすると、救済レーン確保と両立しにくい。
- そのため、`1スレッド相当` は低負荷モードの意味として扱う余地を残すこと。
- 実装時は、速度最適化だけでなく、詰まり回避と救済処理の開始保証を優先して評価すること。
- エンジン外部化を見据え、高負荷検知ロジックはエンジン内に閉じ込めず、上位制御またはサービス連携で扱えるようにすること。
- IPC導入時も、管理者権限サービス未接続を異常終了条件にせず、縮退運転へ落とせる設計を優先すること。
- `Watcher` 側の `AdminUsnMft` とサムネイル側の Disk温度取得で、別々の昇格経路や別々のIPCを作らないこと。
