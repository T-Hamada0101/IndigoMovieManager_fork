# 設計メモ: レーン分類・実行枠予約・Windowsプライオリティ整理（2026-03-06）

更新反映: 2026-03-08

## 1. 目的
- サムネイル処理で混同しやすい「レーン分類」「実行枠予約」「Windowsプライオリティ」を別軸として整理する。
- `slow` と将来の `idle` を検討する際に、どの層へ何を入れるべきかを明確にする。
- 設計上の理想と、2026-03-08 時点の現行実装を分けて読めるようにする。

## 2. 結論
- レーン分類は「ジョブの意味付け」である。
- 実行枠予約は「その種類のジョブに開始機会を先に確保するか」である。
- Windowsプライオリティは「OSへどう譲るか」である。
- `Recovery` は現行実装ではサイズレーンではなく、再試行ジョブに付く実行属性として扱われている。
- 低干渉化を名乗るには、分類と予約と優先度の3つに加えて、UI更新とプレビュー負荷も別段で抑える必要がある。

## 3. 3軸の役割一覧

| 軸 | 役割 | 主な入力 | 主な出力 | 現行実装の主な実体 |
|---|---|---|---|---|
| レーン分類 | ジョブの性質を決める | 動画サイズ, 設定閾値 | `Priority` / `Normal` / `Slow` | `ThumbnailLaneClassifier` |
| 実行枠予約 | 専用席を確保するか決める | backlog, 需要有無, 現在並列数 | 通常枠数, Recovery専用枠, Slow専用枠 | `ThumbnailQueueProcessor` の lease 予約と gate |
| Windowsプライオリティ | OSスケジューラへの優先度を決める | 実行中かどうか, 環境変数, 将来のプリセット設計 | `Normal` / `BelowNormal` / `Idle` 候補 | `ThumbnailQueueProcessor` の `ProcessPriorityClass` 制御 |

## 4. レーン分類の整理

### 4.1 意味
- レーン分類は、ジョブを小動画・通常動画・巨大動画へ分ける段階である。
- 2026-03-08 時点の現行実装では、サイズ分類は `Priority` / `Normal` / `Slow` の3種類である。
- `Recovery` は分類器の出力ではなく、`AttemptCount > 0` で判定される再試行属性である。

### 4.2 現行の分類と属性

| 種別 | 主な判定条件 | 役割 |
|---|---|---|
| `Priority` | 優先レーン閾値以下の小動画 | 短時間ジョブを先に流し、体感スループットを上げる |
| `Normal` | 上記以外かつ巨大動画未満 | 通常処理の主レーン |
| `Slow` | 低速レーン開始閾値以上の巨大動画 | 長時間ジョブを後段へ寄せ、通常動画の停滞を減らす |
| `Recovery` 属性 | `AttemptCount > 0` | 再試行ジョブを通常処理から分けて扱う |

### 4.3 注意
- 分類だけでは、まだ専用枠は確保されない。
- `Slow` と判定されても、予約や専用 gate が無ければ通常枠を消費して動く。
- `Recovery` 属性が付いても、予約が無ければ開始機会は保証されない。

## 5. 実行枠予約の整理

### 5.1 意味
- 実行枠予約は、特定種別のジョブへ専用席を先に回すかを決める段階である。
- 主目的は、巨大動画や再試行動画が通常ジョブに埋もれて飢餓するのを防ぐことにある。

### 5.2 現行の予約方針

| 予約対象 | 発火条件 | 予約内容 | 狙い |
|---|---|---|---|
| Recovery枠 | 再試行需要あり かつ 現在並列数が2以上 | Recovery用に1枠確保 | 再試行開始機会を保証する |
| Slow枠 | 巨大動画需要あり かつ 現在並列数が3以上 | Slow用に1枠確保 | 巨大動画が通常枠を塞ぐのを防ぐ |
| 通常枠 | 上記予約後の残り | 通常ジョブへ割当 | 全体スループットを維持する |

### 5.3 注意
- 予約は需要ベースで有効化する。
- 需要が無い時まで専用枠を固定すると、全体スループットを落とす。
- 現行実装では lease 取得段階で先行確保し、実行時には `DynamicParallelGate` で専用枠を分けている。

## 6. Windowsプライオリティの整理

### 6.1 意味
- Windowsプライオリティは、OSへ「この処理へどれだけ積極的にCPU時間を配るか」を伝える軸である。
- これは分類でも予約でもなく、同じジョブをどれだけ邪魔しにくく走らせるかの話である。

### 6.2 現行実装

| 項目 | 現在の状態 |
|---|---|
| 対象 | サムネイルキュー処理中の現行プロセス |
| 既定値 | `BelowNormal` |
| 切替方法 | 環境変数 `IMM_THUMB_PROCESS_PRIORITY` で上書き可能 |
| 復帰 | キューが空になったら元の優先度へ戻す |
| 制約 | UIと同一プロセスのため、既定では `Idle` まで落としていない |

### 6.3 注意
- 優先度を下げても、UI更新頻度が高ければカーソル引っかかりは残る。
- 逆に、UI更新だけ抑えても CPU / I/O の優先度が高いままだと、裏で重さが残る。
- そのため、最低干渉モードは「優先度低下」だけでなく「UI更新間引き」「プレビュー抑制」「保存並列抑制」も必要である。

## 7. `slow` と `idle` の位置付け

### 7.1 `slow` の現行実装

| 項目 | 現行 |
|---|---|
| 並列数 | `ResolveParallelism` は `2` |
| ポーリング | 基本値の2倍 |
| バッチ間待機 | 750ms |
| 動的復帰 | 無効 |
| レーン予約 | 需要があれば有効 |
| Windowsプライオリティ | 実質 `BelowNormal` 既定 |

### 7.2 `slow` の設計上の意味
- `slow` は「進捗や救済を残しつつ、通常より明確に干渉を減らすモード」として扱う。
- ただし現行は 2 並列のため、設計イメージの「1スレッド相当」とはまだ一致していない。

### 7.3 `idle` の設計目標

| 項目 | `idle` 候補 |
|---|---|
| 主目的 | 最小干渉 |
| 通常並列 | 1本相当 |
| Recovery/Slow予約 | 極力抑える |
| UI更新 | 強く間引く |
| プレビュー | 原則停止 |
| Windowsプライオリティ | `Idle` 候補 |

## 8. 現行実装とのギャップ

| 観点 | できていること | まだ足りないこと |
|---|---|---|
| 分類 | サイズ分類3種はある | `Recovery` は分類器ではなく属性なので、文書上も分けて扱う必要がある |
| 予約 | Recovery / Slow の需要ベース予約はある | `slow` を低干渉モードとしてどう使い分けるかは未整理 |
| Windows優先度 | `BelowNormal` のプロセス優先度低下はある | プリセット連動の段階分けと `Idle` の正式運用は未実装 |
| UI更新 | `RequestThumbnailProgressSnapshotRefresh()` で多重起動は束ねている | 時間ベースの間引きは無い |
| UI反映 | 完了後に `Dispatcher.InvokeAsync` でUI更新している | バッチ化や遅延反映が無い |
| プレビュー | 完了画像プレビューを保持できる | `slow` / `idle` 専用の抑制が無い |

## 9. 実装判断時のルール
- 「巨大動画を `Slow` と判定した」だけでは、低速レーン設計は成立していない。
- 「Recovery専用 gate を作った」だけでは、需要が無い時のスループット最適化は成立していない。
- 「`ProcessPriorityClass` を下げた」だけでは、最低干渉モードは成立していない。
- 最低干渉モードを名乗るには、少なくとも以下を満たす必要がある。
  - 分類が維持されること
  - 予約方針が明確であること
  - Windowsプライオリティが低下していること
  - UI更新頻度が抑制されていること
  - プレビューや保存並列などの体感阻害要因が抑制されていること

## 10. 現行コードとの対応先
- 分類: `src/IndigoMovieManager.Thumbnail.Queue/ThumbnailLaneClassifier.cs`
- プリセット解決: `Thumbnail/ThumbnailThreadPresetResolver.cs`
- 予約と実行 gate: `src/IndigoMovieManager.Thumbnail.Queue/ThumbnailQueueProcessor.cs`
- UI更新要求: `MainWindow.xaml.cs`
- 完了後の UI 反映とプレビュー登録: `Thumbnail/MainWindow.ThumbnailCreation.cs`

## 11. 結論
- レーン分類は「ジョブの意味付け」。
- 実行枠予約は「席の確保」。
- Windowsプライオリティは「OSへの譲り方」。
- 2026-03-08 時点では、この3軸は概ね分離され始めているが、`slow` を本当に低干渉にするには UI 側の抑制設計がまだ必要である。
