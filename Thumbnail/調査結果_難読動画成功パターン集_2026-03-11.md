### P-03. 超短尺・1フレーム動画は `EOF` 後 drain で救える
- 代表:
  - `E:\_サムネイル作成困難動画\画像1枚あり顔.mkv`
  - `E:\_サムネイル作成困難動画\画像1枚ありページ.mkv`
- 初期症状:
  - `autogen failed`
  - `No frames decoded`
  - `receive_frame == EAGAIN` のまま `av_read_frame == EOF`
- 成功の核:
  - `send_packet` 後に即 `EOF` になっても、`avcodec_send_packet(..., null)` で flush して drain すると遅延出力が取れる
- 本体反映:
  - `FfmpegAutoGenThumbnailGenerationEngine` に `EOF` 後の flush drain を追加済み
  - ベンチ経路でも runtime log を見えるようにして再現確認済み
- 関連文書:
  - [サムネイルが作成できない動画対策.md](./サムネイルが作成できない動画対策.md)
  - [引き継ぎ_autogenEOFドレイン対応とベンチ画像出力_f3fd039_2026-03-11.md](./Engines/引き継ぎ_autogenEOFドレイン対応とベンチ画像出力_f3fd039_2026-03-11.md)

### P-04. 短尺動画の先頭黒フレームは「要求秒より前の decoded frame を捨てる」と救える
- 代表: `E:\_サムネイル作成困難動画\_steph__094110-vid1.mp4`
- 観測条件:
  - `duration=3.274014`
  - `0.000s` と `0.033s` は黒
  - `0.100s`, `1.000s`, `2.000s` は明るい
  - 修正前 `autogen` は `Autogen produced a near-black thumbnail`
- 成功の核:
  - backward seek 後に返る古い keyframe をそのまま採用せず、`requested_sec` より十分手前の `decoded_pts_sec` を捨てる
- 本体反映:
  - `FfmpegAutoGenThumbnailGenerationEngine` に `ShouldAcceptDecodedFrameAtRequestedSecond(...)` を追加
  - `decoded_pts_sec` が要求秒から外れすぎる場合は採用せず、次フレームを待つ
- 確認結果:
  - 2026-03-11 09:29:54 に `autogen success`
  - 同日の再実行で `ffmpeg1pass` も success、出力 JPEG 実体も確認

### P-05. `autogen` が `No frames decoded` でも `ffmpeg1pass` 単独成功する群がある
- 代表: `E:\_サムネイル作成困難動画\真空エラー2_ghq5_temp.mp4`
- 観測条件:
  - `duration=5.799`
  - `1440x2560`
  - `avg_frame_rate=120/1`
  - `pix_fmt=unknown`

  - `ffmpeg1pass` は成功し、実 JPEG を保存できる

  - `ffmpeg1pass success`

  - `pix_fmt=unknown` や特殊メタ付き動画は、この群の代表候補

### P-08. index 再構成で全編復旧しなくても、先頭有効区間だけ remux 救出できる群がある
- 代表:
  - `インデックス破壊-093-2-4K.mp4`
- 観測条件:
  - 元動画は `duration=480.947`
  - `ffmpeg` の PNG 抽出は `10/30/60/70/75秒` で成功
  - `80秒以降` は失敗
  - 入力後シークでは `Invalid NAL unit size` と `Decoding error: Invalid data found when processing input`
  - `autogen` は `near-black`
  - `ffmpeg1pass` も元動画では失敗
- remux 結果:
  - `ffmpeg -c copy -movflags +faststart` は完走する
  - ただし出力は `480.947秒` ではなく `76.276秒` に縮む
  - remux 中に `Missing key frame while searching for timestamp`、`Cannot find an index entry before timestamp`、`Packet corrupt` を確認
- 成功の核:
  - これは「index だけ壊れていて再構成すれば全編戻る」ではない
  - 実際には破損点までの先頭有効区間だけを再パッケージでき、その短縮版では `autogen` と `ffmpeg1pass` の両方が成功する
- 含意:
  - `repair/remux` は全編復旧だけを期待せず、「読める範囲だけでも救出してサムネを得る」用途として有効
  - 元動画の中間時刻が壊れていても、先頭有効区間へ縮退させた remux 版なら通常のサムネ経路へ戻せることがある
- 確認結果:
  - `インデックス破壊-093-2-4K.remux.mp4` では `autogen success`
  - 同 remux 版では `ffmpeg1pass success`
  - 動画フォルダに `インデックス破壊-093-2-4K.remux.#64b19f3d.jpg` を出力済み

### F-01. 超短尺で末尾だけ明るい動画は、短尺 fallback 候補が早すぎると救えない
- 代表:
  - `【ライブ配信】神回scale_2x_prob-3.mp4`
- 観測条件:
  - `duration=0.166667`
  - `nb_frames=5`
  - `2160x3840`
  - `autogen failed, Autogen produced a near-black thumbnail`
  - `ffmpeg1pass failed, ffmpeg one-pass failed`
- フレーム実体:
  - `0 / 0.001 / 0.01 / 0.033秒` は平均輝度ほぼ `0.02`
  - `0.1秒` は平均輝度 `116.15`
  - つまり先頭数フレームは黒く、終盤だけ明るい
- 失敗の核:
  - `autogen` は `requested_sec=1/2/3` に対して `decoded_pts_sec=0.033 / 0.066 / 0.1 / 0.133 / 0.166` を受け取れている
  - ただし全フレームが要求秒より手前なので `seek frame skipped` になる
  - 短尺 fallback は `0.001 / 0.01 / 0.016 / 0.033` を試すが、明るい `0.1秒` を踏まない
  - `ffmpeg1pass` も同様に先頭寄り候補しか見ておらず失敗している可能性が高い
- 含意:
  - `_steph__094110-vid1.mp4` のような「先頭黒フレーム」問題の亜種だが、こちらは動画全体が `0.166秒` と極端に短い
  - 「要求秒に近いフレームを優先する」既存ルールだけでは、終盤 1 枚だけ明るい超短尺動画を救えない
- 対応方針:
  - 本体修正は別エージェントへ委譲する
  - 候補:
    - `autogen` の短尺 fallback 候補へ `0.1` を追加
    - `duration <= 0.2s` では「要求秒との距離」ではなく「取れた最終明フレーム」を優先する別ルールを検討
    - `ffmpeg1pass` 側も短尺候補秒を `0.1` まで広げる

## 4. 成功パターンから引ける実装ルール
- `codec` や拡張子だけで失敗群扱いしない。実際に見るべきなのは `movieinfo-probe` の返り方と decode 挙動。
- `No frames decoded` は「フレームが存在しない」とは限らない。超短尺では `EOF` 後 drain が必要。
- `near-black` は「動画が黒い」とは限らない。要求秒より前の古い seek フレーム混入を先に疑う。
- `autogen` 単体調査で `false` が出ても、すぐ動画失敗と断定しない。`IMM_FFMPEG_EXE_PATH` が `ffmpeg.exe` 側を指しているだけで、`tools\ffmpeg-shared` を見ていない誤調査がある。
- `autogen` の単体確認では、`IMM_FFMPEG_EXE_PATH='C:\Users\na6ce\source\repos\IndigoMovieManager_fork\tools\ffmpeg-shared'` を先に固定してから成否を見る。
- ただし `duration <= 0.2s` 級の超短尺では、先頭黒フレーム回避だけでは足りず「終盤 1 枚だけ明るい」失敗固定例もある。
- 長尺 `No frames decoded` でも、`ffmpeg midpoint success` が確認できる群は `autogen` 本体改善とは別に `retry policy` へ逃がす余地がある。
- 超短尺 `No frames decoded` 群でも、`ffmpeg1pass` の極小 seek でだけ取れる個体がある。`EOFドレイン` と seek 候補調整は別の論点として扱う。
- 長尺破損群は `autogen` の精度改善対象と、repair / onepass の責務を分けて扱う。
