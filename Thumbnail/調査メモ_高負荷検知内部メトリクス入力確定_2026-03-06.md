# 調査メモ: 高負荷検知の内部メトリクス入力確定（2026-03-06）

## 1. 目的
- `LTH-009` として、サービス未接続でも使える内部メトリクス入力を確定する。
- `LTH-010` の `HighLoadScore` 仮実装で、何を採用し何を採用しないかを明確にする。

## 2. 結論
- 第1段階の内部メトリクスは、`QueueProcessor` と `ThumbnailEngineRuntimeStats` で取得できる値だけを採用する。
- UIメーター用 `PerformanceCounter` 値は、サムネイル制御の一次入力には採用しない。
- 理由は、UI依存を避けたいこと、取得失敗や無効化の影響を制御本体へ持ち込みたくないこと、将来のオーケストレータ分離時に持ち出しにくいことの3点である。

## 3. 採用する内部メトリクス

### 3.1 Error系
- `batchFailedCount`
  - 現在バッチで失敗したジョブ数。
- `engineSnapshot.AutogenTransientFailureCount`
  - 一時失敗回数。
- `engineSnapshot.FallbackToFfmpegOnePassCount`
  - 1pass フォールバック回数。
- `engineSnapshot.AutogenRetrySuccessCount`
  - 一時失敗後に救済できた件数。

### 3.2 Backlog系
- `activeCountAfterBatch`
  - QueueDB上の未完了件数。
- `currentParallelism`
  - 実効並列数。
- `configuredParallelism`
  - プリセットまたは手動設定から解決された並列数。
- `hasRecoveryDemand`
  - 再試行ジョブが待機または実行中か。
- `hasSlowDemand`
  - 巨大動画ジョブが待機または実行中か。

### 3.3 Throughput系
- `batchProcessedCount`
  - バッチ処理件数。
- `batchMs`
  - バッチ経過時間。
- `totalProcessedCount`
  - 累計処理件数。
- `totalElapsedMs`
  - 累計処理時間。

## 4. 第1段階で採用しない入力

### 4.1 UI依存メトリクス
- CPUメーター値
- GPUメーター値
- HDDメーター値
- 理由:
  - `MainWindow` の UI 更新系で取得しており、オーケストレータ本体の責務ではない。
  - `PerformanceCounter` は環境差と取得失敗があり、制御入力の一次ソースにするには不安定。

### 4.2 管理者権限が必要な入力
- Disk温度
- SMART系
- UsnMft状態
- 理由:
  - 将来の管理者権限サービス側入力として扱うべきで、内部メトリクス版へ混ぜない。

### 4.3 現時点では未整備の入力
- 最古待機時間
- 失敗理由の I/O / timeout 詳細分類
- QueueDB上の recovery 件数、slow 件数の厳密件数
- 理由:
  - 実装追加で取得可能だが、`LTH-009` 時点では未整備。
  - `LTH-010` はまず presence と比率ベースで成立させ、その後必要なら拡張する。

## 5. 第1段階スコアへの割り当て

### 5.1 ErrorScore
- 入力:
  - `batchFailedCount`
  - `AutogenTransientFailureCount`
  - `FallbackToFfmpegOnePassCount`
- 補正:
  - `AutogenRetrySuccessCount` は減点方向の補正に使う。

### 5.2 QueuePressureScore
- 入力:
  - `activeCountAfterBatch`
  - `currentParallelism`
  - `configuredParallelism`
- 目的:
  - 並列数に対して backlog がどれだけ積み上がっているかを見る。

### 5.3 SlowBacklogScore
- 入力:
  - `hasSlowDemand`
- 第1段階:
  - 真偽値ベースの簡易加点とする。

### 5.4 RecoveryBacklogScore
- 入力:
  - `hasRecoveryDemand`
- 第1段階:
  - 真偽値ベースの簡易加点とする。

### 5.5 ThroughputPenaltyScore
- 入力:
  - `batchProcessedCount`
  - `batchMs`
- 目的:
  - backlog が伸びていなくても、1バッチが極端に長い状態を検知する。

## 6. 第1段階の実装方針
- `HighLoadScore` は次の5入力へ整理する。
  - `ErrorScore`
  - `QueuePressureScore`
  - `SlowBacklogScore`
  - `RecoveryBacklogScore`
  - `ThroughputPenaltyScore`
- 第1段階では `MemoryScore` と `CpuScore` を外し、UI非依存で成立させる。
- その代わり backlog と throughput を重めに扱う。

## 7. 次タスクへの引き継ぎ
- `LTH-010`
  - 上記5入力で `HighLoadScore` 仮実装を行う。
- `LTH-011`
  - 段階復帰とヒステリシスを入れる。
- `LTH-017` 以降
  - Disk温度、UsnMft状態、将来の負荷スナップショットを外部入力として合流させる。
