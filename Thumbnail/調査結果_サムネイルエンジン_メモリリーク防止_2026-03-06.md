# 調査結果: サムネイルエンジン メモリリーク防止 2026-03-06

## 結論
- 主因は `Bitmap`/`Mat` の未解放ではなく、長時間運転で増え続ける静的辞書でした。
- 特に `ThumbnailCreationService.cs` の `OutputFileLocks` は、動画ごとの出力先パスを永続保持しており、件数分だけ増加する構造でした。
- 併せて `MainWindow.ThumbnailQueue.cs` の `recentEnqueueByKeyUtc` も、短命用途なのに期限切れ掃除が無く、監視対象が多い運用で徐々に膨らむ状態でした。

## 確認結果

### 1. 高優先: 出力ロック辞書の肥大化
- 対象: `ThumbnailCreationService.cs`
- 症状:
  - `OutputFileLocks` は `saveThumbFileName` ごとに `SemaphoreSlim` を `ConcurrentDictionary` へ追加。
  - 生成完了後は `Release()` のみで、辞書から削除されない。
  - 長時間運転時、処理したユニーク動画数に比例して辞書が増え続ける。
- 対応:
  - 参照カウント付き `OutputFileLockEntry` へ変更。
  - 最終利用者が抜けたタイミングで辞書から除去し、`SemaphoreSlim.Dispose()` まで行う。

### 2. 中優先: デバウンス辞書の肥大化
- 対象: `MainWindow.ThumbnailQueue.cs`
- 症状:
  - `recentEnqueueByKeyUtc` は 800ms の重複抑止専用。
  - しかし古いキーを掃除しておらず、監視イベントのキーが累積する。
- 対応:
  - 一定回数ごとに期限切れキーを prune する処理を追加。
  - 実用上必要な時間窓を超えたエントリだけ削除する。

### 3. 低優先: 画像/デコーダ本体の解放
- 対象:
  - `Engines/FfmpegAutoGenThumbnailGenerationEngine.cs`
  - `Engines/FrameDecoderThumbnailGenerationEngine.cs`
  - `Decoders/FfMediaToolkitThumbnailFrameDecoder.cs`
  - `Decoders/OpenCvThumbnailFrameDecoder.cs`
- 確認内容:
  - `Bitmap` は `using` または `finally` で概ね解放されている。
  - FFmpeg.AutoGen の `AVFrame` / `AVPacket` / `AVCodecContext` / `AVFormatContext` / `SwsContext` も `finally` で解放されている。
  - OpenCV の `Mat` も `using` で閉じている。
- 判断:
  - 現時点の一次原因はここではなく、静的保持オブジェクトの肥大化側。

## 追加した回帰確認
- `Tests/IndigoMovieManager_fork.Tests/ThumbnailMemoryLeakPreventionTests.cs`
- `CreateThumbAsync` 実行後に、出力ロック辞書件数が 0 に戻ることを確認するテストを追加。

## 補足
- `MovieMetaCache` は上限到達時に `Clear()` されるため、無制限肥大化の構造ではありません。
- `ThumbnailPreviewCache` も LRU 上限付きのため、今回の主因からは外しました。
