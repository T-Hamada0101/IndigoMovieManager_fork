# AI向け サムネ成功後メインタブ再読込ガイド 2026-03-30

最終更新日: 2026-06-18

変更概要:
- サムネ成功 / rescued sync の選択中反映は、汎用 `Refresh()` ではなく `RefreshSelectedThumbnailDetail()` へ寄せた。タグ編集再表示を巻き込まず、選択中詳細のサムネ表示だけを再評価する。
- サムネ成功後に main tab の画像が変わらない時の正本を追加
- 即時反映と `Reload` 相当再構築の使い分けを明文化
- 今後 AI が様々な導線から再利用できる判断基準を整理
- preferred 成功でも対象 `MovieRecords` へ直接反映できた場合は、後段の `Reload` 相当再構築を省く方針へ更新
- manual rescue 即時反映でも対象 `MovieRecords` へ直接反映できた場合は、後段の `Reload` 相当再構築を省く方針へ更新
- 対象 `MovieRecords` へ直接反映できない時の後段も `FilterAndSort(..., true)` ではなく、visible refresh / preferred key revision / snapshot 予約へ寄せる方針へ更新

## 1. この資料の目的

- サムネ成功後の main tab 再読込まわりを AI が迷わず触れるようにする
- `path 差し替え` と `Reload 相当再構築` の役割差を固定する
- 「成功したのに画像が変わらない」時に、どこを見るかを共通化する

## 2. 現在の結論

- 軽い成功反映だけで済む場合:
  - `TryApplyThumbnailPathToMovieRecord(...)`
  - `RefreshVisibleThumbnailUiAfterImmediateThumbnailSuccess(...)`
- それでも main tab が変わらない場合、または対象 `MovieRecords` へ直接反映できない場合:
  - `RequestMainTabLocalRefreshAfterThumbnailSuccess(...)`
  - 後段で失敗キャッシュ無効化、visible refresh、preferred key revision、ERROR/進捗 snapshot 予約を1回だけ流す
  - サムネERROR順だけは順序が変わるため、DB 再読込ではなく `SortDataAsync("28")` で現在一覧を並べ直す

今回の実機確認では、
- 軽い再描画だけでは main tab が変わらないケースがある
- ヘッダーの `Reload` ボタンなら更新される

このため、今の正本は
- まず軽い即時反映
- 次に短いデバウンス付きの局所 refresh

の二段構えである。

## 4. 処理の流れ

### 4.1 通常の成功反映

入口:
- `CreateThumbAsync(...)`

主処理:
- `TryApplyThumbnailPathToMovieRecord(...)`
- `TryReflectRescuedThumbnailIntoUpperTabRescueItems(...)`
- `TryReflectCreatedThumbnailIntoUpperTabDuplicateItems(...)`
- `ShouldRequestMainTabLocalRefreshAfterThumbnailSuccess(...)`

役割:
- まず対象 `MovieRecords` の `ThumbPath*` を差し替える
- 同一パス上書きでも空経由で再通知する
- 特殊タブ側のコピー済み ViewModel も合わせて差し替える

### 4.2 軽い visible UI 再評価

入口:
- `RefreshVisibleThumbnailUiAfterImmediateThumbnailSuccess(...)`

役割:
- `InvalidateThumbnailErrorRecords(refreshIfVisible: true)`
- `RefreshSelectedThumbnailDetail()`
- `RequestUpperTabVisibleRangeRefresh(immediate: true, reason: ...)`
- `RequestThumbnailErrorSnapshotRefresh()`
- `RequestThumbnailProgressSnapshotRefresh()`

用途:
- 即時に見た目を追従させたい時の第一段
- manual rescue の即時成功
- preferred なユーザー明示作成成功

### 4.3 局所 refresh の後段再評価

入口:
- `RequestMainTabLocalRefreshAfterThumbnailSuccess(...)`

内部:
- `DispatcherTimer` で短く圧縮
- `ThumbnailSuccessMainTabReloadTimer_Tick(...)`
- `RefreshMainTabLocallyAfterThumbnailSuccessAsync(...)`

役割:
- DB 再読込ではなく、表示中の画像 Binding と下部 snapshot を再評価する
- 複数成功が連続しても最新1回へ圧縮する
- `path 差し替え` だけで届かない main tab の表示更新を拾う
- preferred 成功や manual rescue 即時反映でも、対象 `MovieRecords` へ直接反映できた場合は forced rebind と visible refresh を優先し、この後段 reload は予約しない
- サムネERROR順だけは成功後に順序が変わるため、現在一覧の in-memory sort だけを許可する

## 5. なぜ full reload ではなく局所 refresh へ寄せるか

過去の観測事実:
- 成功後に画像が変わらない
- ただしヘッダー `Reload` ボタンなら更新される

当時はこの事実から、
- 単なる `PropertyChanged`
- 軽い `RefreshSelectedThumbnailDetail()`
- visible range 再評価

だけでは足りず、実際には
- 一覧ソース再構築
- サムネパス再解決

まで含む `Reload` 系の筋が効いていると判断した。

2026-06-18 時点では、サムネ成功は WhiteBrowser DB を変更しないため、main tab の後段 `FilterAndSort(..., true)` は通常経路から外す。選択中詳細の再評価も汎用 `Refresh()` ではなく `RefreshSelectedThumbnailDetail()` に限定し、タグ編集再表示を巻き込まない。直接反映できなかった行も、失敗キャッシュ無効化、visible refresh、preferred key revision、下部 snapshot 予約で画像 Binding を再評価する。唯一、サムネERROR順だけは成功で並び順が変わるため、DB 再読込ではなく現在一覧の `SortDataAsync("28")` に限定する。

## 6. どの導線で使うか

使う:
- `等間隔サムネイル作成` のようなユーザー明示作成
- 今後増える「ユーザーが結果をすぐ見たい単動画処理」

慎重に使う:
- 対象 `MovieRecords` へ直接反映済みの preferred 成功
- 対象 `MovieRecords` へ直接反映済みの manual rescue 成功
- 自動 watch 起点の大量成功
- バックグラウンドで連続発火する bulk 成功

理由:
- `FilterAndSort(..., true)` はヘッダー `Reload` 相当で比較的重い
- 自動系へ無差別に入れるとテンポを壊す

## 7. AI が守るべきルール

- まずは `TryApplyThumbnailPathToMovieRecord(...)` を壊さない
- 次に軽い即時反映 helper を使う
- それでも main tab が追従しない時、または対象行へ直接反映できない時だけ `RequestMainTabLocalRefreshAfterThumbnailSuccess(...)` を使う
- 自動系 bulk 成功へ安易に `FilterAndSort(..., true)` を直書きしない
- 新しい成功導線を増やしたら、この資料と `Thumbnail/README.md` を同時更新する

## 8. 実装上の注意

- `ThumbPathBig10` は `PropertyChanged` 名ミスがあると更新が飛ばない
- 同一 jpg の上書きでは、空経由の forced rebind が必要
- `RequestMainTabLocalRefreshAfterThumbnailSuccess(...)` は UI スレッドで扱う。非 UI スレッドから来た時は `DispatcherPriority.Background` で戻し、shutdown 中は予約しない
- timer は最新1回に圧縮する前提で使う

## 9. テスト観点

最低限:
- `TryApplyThumbnailPathToMovieRecord(...)` が同一パスでも2回通知する
- `ThumbPathBig10` が正しい property 名で通知する
- `ShouldRefreshVisibleThumbnailUiAfterCreate(...)` が preferred のみ true
- `ShouldRequestMainTabLocalRefreshAfterThumbnailSuccess(...)` が、preferred / manual rescue 相当でも直接反映済みなら false を返す
- `RequestMainTabLocalRefreshAfterThumbnailSuccess(...)` の非 UI スレッド経由が `DispatcherPriority.Background` と shutdown guard を通る
- `RefreshVisibleThumbnailUiAfterImmediateThumbnailSuccess(...)` と rescued sync 完了時に、汎用 `Refresh()` へ戻らず `RefreshSelectedThumbnailDetail()` へ進む
- `ThumbnailSuccessMainTabReloadTimer_Tick(...)` が `FilterAndSort(..., true)` へ戻っていない

実機観点:
- main tab で成功直後に画像が変わるか
- 少し遅れてでも `Reload` 相当で最終的に追従するか
- 連続成功時に過剰な全面再読込になっていないか

## 10. 迷った時の判断

- 「軽く直る」なら軽い即時反映で止める
- 「Reload でしか直らない」ように見えても、まず局所 refresh と preferred key revision で届くか確認する
- 「自動系にも入れたい」時は、まずテンポ悪化リスクをレビューする

今後 AI がこの領域を触る時は、この資料を正本として更新してからコードを変えること。
