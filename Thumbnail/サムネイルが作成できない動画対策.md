# サムネイルが作成できない動画対策

最終更新: 2026-03-10

## 1. 結論
- 「サムネイルが作れない動画」は1種類ではなく、少なくとも次の5系統に分かれる。
- アプリ側で吸収できるものは実装修正で救済し、実フレーム自体が読めないものだけを最終失敗として扱うのが妥当。
- 2026-03-09 時点で、主な実装修正はすでに投入済みである。

## 2. 主な失敗パターン

### 2.1 コンテナのインデックスやメタ情報が壊れている
- 映像自体は読めるが、シーク位置やフレーム位置の管理情報が壊れているケース。
- 典型例:
  - `mp4`, `m4v`, `3gp`, `3g2`
  - `mov`
  - `avi`, `divx`
  - `mkv`
  - `flv`, `f4v`
  - `wmv`, `asf`
  - `mts`, `m2ts`
- この系統は「再構成すれば読める」ことが多い。

### 2.2 コンテナ尺が壊れている
- 特に `AVI` で、実動画は約80秒なのにコンテナ上の再生時間だけが数時間に化けることがある。
- この状態だと、サムネイル取得時刻の計算が後ろへ飛び過ぎて、フレーム取得に失敗しやすい。

### 2.3 1秒未満の極短尺動画
- 例: `0.033秒`, `0.069秒`
- 従来は整数秒ベースの時刻計算で全パネルが `0秒` に潰れ、オートジェン経路で取りこぼしやすかった。
- 実フレームが存在しても、取得位置の作り方だけで失敗していた。

### 2.4 監視や索引の問題で「動画が見えていない」
- `Everything` の索引未更新、空結果、パス条件差で検出漏れするケース。
- 動画そのものは正常でも、スキャン段階で候補に入らないとサムネイル処理へ進まない。

### 2.5 絵文字パスで表示側の再取得に失敗する
- サムネイル生成自体は成功していても、UI が `movie_path` で DB を引き直す箇所で一致に失敗すると一覧へ反映されない。
- 特に `lower(movie_path)` のような変換付き比較や、SQL 文字列埋め込みは絵文字入りパスと相性が悪い。
- 例: `D:\BentchItem_HDD\→✌🏻🍒\IMG_1346.MOV`

### 2.6 実フレームが本当に読めない
- `ffprobe` では stream 情報が見えても、`ffmpeg` の1フレーム抽出で失敗するケースがある。
- この系統はアプリ側のシーク補正では救えない。

## 3. 実装済み対策

### 3.1 リカバリー対象拡張子の拡大
- 従来は `flv` だけをインデックス再構成対象にしていた。
- 現在は、主要なインデックス持ちコンテナ全体へ対象を拡大済み。
- これで「2回失敗したら recovery に入り、再構成を試す」範囲が実態に近くなった。

### 3.2 2回失敗した動画は必ず recovery に残す
- 再スキャンや再投入で `AttemptCount` が 0 に戻ると、永遠に通常処理を回り続ける。
- これを防ぐため、`AttemptCount >= 2` の既存行は失敗回数と `LastError` を保持するよう修正済み。

### 3.3 `AVI` の壊れた再生時間は映像実尺を優先
- コンテナ尺と `総フレーム数 / FPS` が大きく乖離する場合は、映像側の実尺を優先する。
- これにより、尺異常でサムネイル時刻が壊れる `AVI` を救済できる。
- デバッグ時は `avi-duration-recovery` で尺補正判定を、`thumbinfo-final` と `autogen-seek` で実際に seek へ渡した秒数を確認する。
- 自動サムネ生成では、`AVI`/`DIVX` の duration 取得元が `request / provider / shell` のどれでも、`BuildAutoThumbInfo` の直前で実尺へ正規化する。
- `out1.avi` では shell duration `29742` が `BuildAutoThumbInfo` に入り、`thumb_sec=[14871]` を作っていた。
- 現在は `ThumbnailJobMaterialBuilder` で `FFMediaToolkit` を優先し、失敗時は `OpenCV` で実尺を再取得してから自動サムネ時刻を作る。
- 取得元と補正有無の切り分けは `thumbinfo-build` の `duration source resolved`、`duration actual probe selected`、`duration corrected before auto thumb build` を見る。
- `MovieInfo.ProbeMetadataSources(path)` で `FFMediaToolkit / OpenCV / autogen` の3系統比較ができる。指定動画の尺差分確認はこの入口を使う。
- 最小実行は DEBUG 時の起動引数 `--movieinfo-probe "C:\path\to\target.avi"` を使う。比較結果はダイアログ表示され、`debug-runtime.log` にも `movieinfo-probe` で残る。

### 3.4 1秒未満動画の取得時刻を実時間で再配置
- オートジェン経路では、`1秒未満` かつ全パネル `0秒` のときだけ、実デコード時刻を動画長の中へ均等配置する。
- JPEG末尾へ埋めるメタ情報の形式は維持しつつ、実際のフレーム取得だけを改善している。

### 3.4.1 `ffmpeg1pass` でも短尺専用 seek 候補を試す
- `ffmpeg1pass` 初回失敗時、`duration <= 1.0秒` かつ `5パネル以下` の動画では、先頭付近の `seek` 候補を浅く再試行する。
- 候補は `0.001, 0.005, 0.01, 0.016, 0.033, 0.05, 0.069, 0.1, 0.25, 0.5秒` と、動画長由来の `10% / 25% / 50% / 終端-0.01秒`。
- 既に試した開始秒と、動画長を超える候補は除外する。
- 既存の engine route や repair 条件は変えず、`ffmpeg1pass` の中だけで短尺救済を強化する。

### 3.5 `autogen` で通常秒が全滅した時は先頭付近を1枚だけ救済
- `ThumbSec` で1枚も取れず `No frames decoded` へ落ちる動画に対しては、`autogen` が先頭付近の候補秒を浅く再探索する。
- 候補は `0, 0.1, 0.25, 0.5, 1, 2秒` を動画長で丸めたもの。
- 1枚でも取れた場合は、その画像を既存タイル数へ複製して保存する。
- デバッグ時は `autogen-header-frame-fallback` を見ると、fallback が `requested / try / hit / miss` のどこまで進んだか分かる。
- ただし、`autogen-header-frame-fallback` のログが追加されていても、最終成功画像が必ずこの経路で作られたとは限らない。
- 短尺かつ破損気味の `mkv` では、`autogen` が `No frames decoded` で落ちた後、repair を挟んで `ffmpeg1pass` fallback が最終取得する例を確認している。
- `autogen-header-frame-fallback`
  - この fallback 専用のログカテゴリ名。通常 seek の `autogen-seek` と切り分けて見る。
- `fallback requested`
  - 通常の `ThumbSec` seek では1枚も取れず、先頭付近 fallback へ入ったことを示す。
  - `duration_sec`, `cols`, `rows`, `thumb_sec` が出るので、どのパネル構成と元 seek 群で救済に入ったか分かる。
- `fallback candidates`
  - 実際に試す候補秒一覧を出す。
  - 動画長で丸めた結果もここで確認できる。
- `fallback try`
  - 候補秒を1件ずつ試し始めたことを示す。
  - どの秒で当たったか、どこまで進んだかを追える。
- `fallback hit`
  - その候補秒で1枚取れたことを示す。
  - この後は通常失敗ではなく、複製保存へ進む。
- `fallback tile replicate`
  - 取れた1枚を既存サムネ形式に合わせて、タイル数ぶん複製したことを示す。
  - `tile_count` で何枚構成へ展開したか分かる。
- `fallback exhausted`
  - 全候補を試しても1枚も取れなかったことを示す。
  - この場合は従来どおり `No frames decoded` 側へ落ちる。

### 3.6 監視フォルダ保存後の即時反映
- 監視フォルダ編集画面を閉じた後、watcher 再構築と `CheckMode.Auto` の再走査を即時実行するよう修正済み。
- 新規DBでフォルダ登録したのに動画が出てこない、という誤解を減らせる。

### 3.7 `Everything` 空結果時の filesystem fallback
- `Auto` と `Manual` の全量走査では、`Everything` が 0 件を返したときに filesystem 走査へフォールバックする。
- これで索引未更新や一時的な空結果でも検出漏れを避けられる。

### 3.8 `Everything` クエリの末尾 `\` を除去
- `"E:\path\" ext:mp4` のような末尾バックスラッシュ付きクエリは、環境によって結果が不安定だった。
- 現在は `"E:\path" ext:mp4` 形式へ揃えている。

### 3.9 絵文字パスは `movie_path` をパラメータ完全一致で引く
- サムネイル生成済みなのに表示されないケースでは、生成側ではなく UI 反映側の `movie_path` 再取得を疑う。
- `Watcher\MainWindow.Watcher.cs` と `Thumbnail\MainWindow.ThumbnailCreation.cs` で、`lower(movie_path) = lower('...')` をやめた。
- 現在は `movie_path = @movie_path` の `SQLiteParameter` 完全一致で取得する。
- これにより、`D:\BentchItem_HDD\→✌🏻🍒\IMG_1346.MOV` のような絵文字フォルダ配下でも、生成済みサムネイルを UI が正しく拾える。
- DB ヘルパー `DB\SQLite.cs` には、パラメータ付き `GetData(...)` を追加済み。

## 4. 実測した6動画の整理

`<difficult-video-root>/作成NG` の6動画を `ffprobe` と `ffmpeg` で確認した。

| No | 形式 | 概要 | 判定 |
|---|---|---|---|
| 01 | mp4 | 3.27秒, H.264, 97 frames | 正常寄り |
| 02 | mp4 | 120fps, `pix_fmt` と `nb_frames` 欠落 | メタ情報が不完全 |
| 03 | mkv | 約0.98秒, `ffmpeg` で1フレーム抽出失敗 | 実フレーム読取不能 |
| 04 | mkv | 0.033秒 | 極短尺だが抽出可能 |
| 05 | mkv | 0.069秒 | 極短尺だが抽出可能 |
| 06 | wmv | 約195秒, WMV3 | 正常寄り |

補足:
- `03` は `Cannot determine format of input 0:0 after EOF` となり、アプリ補正では救えない。
- `04` と `05` は壊れているのではなく、短尺すぎて時刻計算の弱さが出ていた系統である。

### 4.1 短尺破損 `mkv` の実測補足 2026-03-10
- 対象は `画像1枚ありページ.mkv` で、実際に生成されたサムネイルは次の2枚。
  - `bin\x64\Debug\net8.0-windows\Thumb\1111\120x90x1x1\画像1枚ありページ.#d6d68a0c.jpg`
  - `bin\x64\Debug\net8.0-windows\Thumb\1111\56x42x5x1\画像1枚ありページ.#d6d68a0c.jpg`
- この2枚はどちらも `Idle worker` が処理し、最終成功エンジンは `ffmpeg1pass` だった。
- つまり、今回の成功は `autogen-header-frame-fallback` で直接取れたのではなく、`autogen` 失敗後の fallback 成功である。
- 時系列は次のとおり。
  - `2026-03-10 08:58:55.830` に `120x90x1x1` を `autogen` で試行し、`failed, No frames decoded`
  - `2026-03-10 08:58:55.983` に `56x42x5x1` を `autogen` で試行し、`failed, No frames decoded`
  - `2026-03-10 08:59:03.792` に `120x90x1x1` を `ffmpeg1pass` で `success`
  - `2026-03-10 08:59:04.302` に `56x42x5x1` を `ffmpeg1pass` で `success`
- 根拠ログ:
  - `%LOCALAPPDATA%/IndigoMovieManager_fork/logs/thumbnail-create-process.csv`
  - `505628 autogen ... 120x90x1x1 ... failed, No frames decoded`
  - `505629 autogen ... 56x42x5x1 ... failed, No frames decoded`
  - `505635 ffmpeg1pass ... 120x90x1x1 ... success`
  - `505636 ffmpeg1pass ... 56x42x5x1 ... success`
  - `%LOCALAPPDATA%/IndigoMovieManager_fork/logs/thumbnail-worker-thumb-idle_*.log`
  - `1689 thumbnail done role=Idle ... 120x90x1x1...`
  - `1697 thumbnail done role=Idle ... 56x42x5x1...`
- さらに直前に `index-repair-summary repair forced` も出ており、この動画は修復テンポラリ経路を通っている。
- したがって、この事例の整理は「autogen では取れなかったが、repair を挟んだ後に `ffmpeg1pass` fallback で最終取得できた短尺破損動画」である。
- `autogen-header-frame-fallback` の有無だけで成功原因を決め打ちせず、必ず `thumbnail-create-process.csv` の最終成功エンジンまで追うこと。

## 5. 失敗時の切り分け手順

### 5.1 まず `ffprobe` で全情報を見る
```powershell
C:\ffmpeg\bin\ffprobe.exe -hide_banner -v quiet -print_format json -show_format -show_streams "対象動画"
```

見る点:
- `duration`
- `nb_frames`
- `pix_fmt`
- `avg_frame_rate`
- stream の有無

### 5.2 次に `ffmpeg` で1フレーム抜けるか試す
```powershell
C:\ffmpeg\bin\ffmpeg.exe -v error -y -i "対象動画" -frames:v 1 "出力.jpg"
```

判定:
- ここで抜けるなら、アプリ側の時刻計算や索引依存の問題を疑う。
- ここで抜けないなら、実フレーム破損の可能性が高い。

### 5.3 インデックス系が怪しい場合は無劣化再構成を優先
```powershell
ffmpeg -i input.mp4 -c copy -map 0 -movflags +faststart output.mp4
```

用途:
- MP4/MOV 系の `moov` 再配置
- ストリーム構造の再パック

### 5.4 それでも駄目なら再エンコード
```powershell
ffmpeg -i input.mp4 -c:v libx264 -pix_fmt yuv420p -g 30 -c:a copy output.mp4
```

用途:
- GOP やキーフレーム配置まで含めて作り直す
- 互換性を優先して救済する

### 5.5 短尺動画の ffmpeg 明示試験ハーネス
- `Tests/IndigoMovieManager_fork.Tests/FfmpegShortClipRecoveryPlaygroundTests.cs` を追加した。
- 既定対象は `<difficult-video-root>/画像1枚ありページ.mkv`。
- 環境変数 `IMM_TEST_FFMPEG_SHORT_MOVIE_PATH` で対象動画を差し替えできる。
- このハーネスでは次を順に試す。
  - `ffmpeg1pass` の `1x1` / `5x1`
  - duration 差し替え付き `ffmpeg1pass`
  - 生 `ffmpeg.exe` のサブ秒 `preseek` / `postseek`
  - repair 後入力への再試行
- 成功試行と失敗試行の両方を `FailureDb` に追記するため、どの seek とどの経路が当たり、どこが外れたかを後から比較できる。
- `scripts/export_failuredb_shortclip_seek_summary.ps1` を追加した。
- 既定では最新の `.failure-debug.imm` から `Done` 行だけを拾い、`engine + seek + status` 単位で集計する。
- さらに `%LOCALAPPDATA%/IndigoMovieManager_fork/logs` 配下の runtime log を読み、`autogen-header-frame-fallback` / `ffmpeg1pass-shortclip-seek` の `fallback hit` / `fallback miss` / `fallback exhausted` を同じ動画パスと近い時刻で突合する。
- 例:
```powershell
pwsh -NoLogo -File .\scripts\export_failuredb_shortclip_seek_summary.ps1
```
- MainDB から引きたい時:
```powershell
pwsh -NoLogo -File .\scripts\export_failuredb_shortclip_seek_summary.ps1 `
  -MainDbPath "C:\path\to\sample.wb"
```
- runtime log の場所や許容秒を変えたい時:
```powershell
pwsh -NoLogo -File .\scripts\export_failuredb_shortclip_seek_summary.ps1 `
  -LogRoot "$env:LOCALAPPDATA\\IndigoMovieManager_fork\\logs" `
  -RuntimeMatchWindowSec 180
```
- `-IncludeFailed` を付けると失敗試行も同時に集計できる。

## 6. 運用方針
- 2回失敗した動画は recovery レーンへ送る。
- recovery ではインデックス再構成対象拡張子に対して修復を試す。
- `ffmpeg` で1フレームも抜けない動画は、無理に通常処理へ戻さず最終失敗として残す。
- `Everything` の結果が怪しい時は、手動更新で filesystem fallback が効いているかログを確認する。

## 6.1 workthree 受領後の反映順
- `workthree` から優先順位表や伝達 doc が返ったら、まず `FailureDb.ExtraJson` に必要キーが揃っているかを確認する。
- 次に `result_signature` を既存 `FailureKind` へ当てはめ、既存分類で足りるかを決める。
- その後、導入位置を `preflight / retry policy / repair workflow / finalizer` から1つ選ぶ。
- 導入位置が決まったら、対応する Implementation Plan へ一般条件だけを追記する。
- 最後に `サムネ失敗` タブ、観測スクリプト、手動確認手順の追随要否を決める。
- 個別動画名は運用記録には残してよいが、本線実装条件には持ち込まない。

## 7. 残課題
- `Watch` 常時監視モードでも、必要なら条件付き filesystem fallback を入れる余地がある。
- 真に壊れた動画を早期に分類するため、`ffmpeg` 1フレーム抽出相当の軽量診断を recovery 前段へ追加する余地がある。
- DRM や保護付き古い `WMV` は、通常のインデックス再構成では救えない場合がある。
- 絵文字パスまわりは生成経路だけでなく、DB 再取得や UI 反映で `movie_path` を比較する箇所を引き続き棚卸しする余地がある。
