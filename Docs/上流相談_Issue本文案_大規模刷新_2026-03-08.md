## タイトル案
大規模刷新の受け入れ可否相談: 監視・サムネイル・キュー・高速化を含む別系統改造

## 本文案
お疲れさまです。

このフォークでは、現行の `IndigoMovieManager` を前提にした局所改善ではなく、監視、サムネイル生成、キュー処理、外部依存の扱いまで跨ぐ大規模刷新を進めています。

最初に結論を書くと、これは通常の小PRへ分解して自然に取り込むタイプの変更ではありません。
そのため、コードPRの前に「この方向性自体を受け入れる意思があるか」を先に相談したいです。

### 何を変えているか
- 監視フォルダ処理を、単純な `FileSystemWatcher` 依存から拡張し、Everything / USN-MFT / 通常ファイルシステムの複数プロバイダを使い分ける構成へ寄せています。
- サムネイル生成を、UI従属の重い処理から外し、キュー駆動・エンジン切替・失敗救済を持つ構成へ寄せています。
- サムネイル系は `autogen` / `ffmediatoolkit` / `ffmpeg1pass` / `opencv` の複数経路を持ち、条件に応じて切替えています。
- Queue / Engine / WorkerCore / ProgressViewer など、責務分離を進めています。

### 狙い
- UIの詰まりを減らす
- 監視とサムネイルのボトルネックを構造で外す
- 絵文字パス、高負荷動画、失敗再試行などの実運用問題に耐える
- WhiteBrowser互換DBは維持する

### 現時点の効果例
- 小ファイル1パネル生成のベンチでは、`autogen` の平均が `110ms`、`ffmediatoolkit` が `226ms`、`ffmpeg1pass` が `342ms` でした
- CRC32は `System.IO.Hashing` 化で、既存値互換を保ったまま旧ロジック比で約5倍高速でした

### 相談したいこと
この刷新について、上流として次のどの扱いを望むでしょうか。

1. 本流採用を前提に、Draft PR または統合ブランチで検討したい
2. まずは実験ブランチ扱いで保持し、効果検証を続けたい
3. 本流では採らず、フォーク側の独立路線としたい

### 補足
- もし 1 または 2 であれば、次に比較資料、互換性一覧、統合単位を整理して提示します
- もし 3 であれば、無理に命名や構成だけ上流へ寄せる作業はせず、フォーク側で独立して育てます

この相談の段階では、細部レビューよりも「この刷新方向を受け入れるかどうか」の意思決定をお願いしたいです。

必要であれば、次に以下をまとめます。
- 旧構成と新構成の差分図
- 互換性維持点 / 非維持点の一覧
- 統合時のリスクとロールバック案
