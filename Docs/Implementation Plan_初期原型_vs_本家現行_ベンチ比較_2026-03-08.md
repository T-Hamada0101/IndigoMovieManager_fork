# 初期原型 vs 本家現行 ベンチ比較計画（2026-03-08）

## 1. 結論
- ベンチは取れる。
- ただし、`このリポジトリの最初のcommit` を `本家同様` と見なしてはいけない。
- 今すぐ固定して測る対象は次の2本である。
  - `709a137` : 初期原型ベースライン
  - `upstream_local` 現行HEAD : 本家現行ベースライン
- 現行フォークは実装中のため、安定後に第2段階で追加する。

## 2. なぜ2本必要か
- ルートコミット `4337abc` / `e48e2ea` は `LICENSE.txt` 追加だけで、アプリ本体ではない。
- 最初の実コード追加 `709a137` は、かなり初期の素の構成である。
- 現在の本家現行は、少なくとも次を持つ。
  - `ThumbnailQueueProcessor`
  - `ThumbnailCreationService`
  - `Thumbnail/MainWindow.ThumbnailCreation.cs`
  - `Thumbnail/MainWindow.ThumbnailQueue.cs`
- したがって、`709a137` は本家現行の代用ではなく、あくまで `原型との比較` に使う。

## 3. 比較対象の定義

### 3.1 Case A: 初期原型
- 対象コミット: `709a137`
- 性格:
  - OpenCvSharp中心
  - `MainWindow.xaml.cs` に `CheckThumbAsync` / `CreateThumbAsync` が直置き
  - `Crc32.NET` ベース
  - FFmpeg.AutoGen / FFMediaToolkit / Everything / Worker分離なし
- 用途:
  - 「最初期からどれだけ進化したか」を測る

### 3.2 Case B: 本家現行
- 対象: `<upstream-repo-root>` の現行HEAD
- 性格:
  - Queue系、ThumbnailCreationService、分離済みサムネイル処理あり
  - ただし現時点ではフォークほどの大改造ではない
- 用途:
  - 「今の本家がどこまで進んでいるか」を測る

### 3.3 Case C: 現行フォーク（第2段階）
- 対象: `<fork-repo-root>` の現行HEAD
- 状態:
  - 現在は実装中のため、ベースラインには使わない
- 用途:
  - 初期原型と本家現行の比較が終わった後に追加する比較対象

## 4. 何を測るか

### 4.1 最低限の共通指標
- アプリ起動時間
- DBオープン時間
- 監視フォルダ初回走査時間
- 新規動画検出から一覧反映までの時間
- サムネイル1件生成時間
- サムネイル複数件バッチ完了時間
- ハッシュ取得時間
- エラー発生率

### 4.2 最重要指標
- 小ファイル1件サムネイル生成
- 中規模フォルダの初回取り込み
- 監視中の動画追加反映
- 同一条件でのCPU使用率とメモリ傾向

## 5. 比較の前提条件
- 同じマシンで測る
- 同じ電源モードで測る
- 同じ動画セットを使う
- 同じDB初期状態を使う
- 初回ウォームアップと本番計測を分ける
- 可能なら各ケース3回以上実行して中央値を見る

## 6. データセット設計
- ベンチ用データセットの固定パス: `D:\BentchItem_HDD`

### 6.1 サムネイル単体
- 小ファイル 10本
- 中サイズファイル 10本
- 長尺ファイル 3本
- 絵文字パスあり 5本
- 壊れ気味動画 2本

### 6.2 監視・初回走査
- 100件規模フォルダ
- 500件規模フォルダ
- サブフォルダ混在
- サムネイル既存あり / なし を分ける

## 7. 実施順

### Step 1: 初期原型を固定する
- `709a137` から作業ブランチを切る。
- 推奨名: `codex/bench/709a137-baseline`
- ここでは機能追加しない。
- 必要なら最小限の計測ログだけ足す。

### Step 2: 本家現行を固定する
- `<upstream-repo-root>` 側で現行HEADを固定する。
- 推奨名: `codex/bench/upstream-current-baseline`
- 本家側に計測ログが足りない場合だけ、ローカル用の計測枝を作る。

### Step 3: まずは2ケースで測る
- A: `709a137`
- B: 本家現行HEAD
- ここで差分傾向を確認する。

### Step 4: フォーク現行は安定後に追加する
- 現行フォークHEADは、実装が一段落してから `codex/bench/fork-current` を切る。
- 追加時は A / B と同じシナリオで再測定する。

## 8. ベンチの取り方

### 8.1 まずは手動計測でよい
- 初回は自動化に時間を使いすぎない。
- まずは以下だけを揃えて測る。
  - 計測開始時刻
  - 計測終了時刻
  - 対象動画数
  - 成功件数 / 失敗件数
  - 例外ログ

### 8.2 自動化は第2段階
- 手動で差が十分に見えたら、PowerShellスクリプト化する。
- 初期版と本家現行では構造が違うので、最初から完全共通スクリプトを狙わない。

## 9. 実務上の注意
- `709a137` には今のベンチ基盤がない。
- そのため、`今のベンチスクリプトをそのまま流す` のは基本できない。
- 必要なら各ケースに `同じ粒度の計測ログ` を足して合わせる。
- 比較の目的は `絶対値の完全一致` ではなく、`同一条件での傾向差` を取ること。

## 10. 期待できるアウトプット
- まずは 初期原型 → 本家現行 の改善幅
- 次に 本家現行 → フォーク現行 の改善幅
- どの改善が `本家に戻したい価値` で、どれが `フォーク専用最適化` かの切り分け

- `最初のcommitだけ見て本家同様比較` はやらない。
- フォーク現行は安定後に第2段階で追加する。
