# PR向け上流整合化 実装計画（2026-03-08）

## 1. 目的
- フォーク版の成果を、フォーク元が受け入れやすい形でPRできる状態へ整える。
- `IndigoMovieManager_fork` 固有の命名・設定・運用差分を整理し、上流へ戻すべきものと残すべきものを分離する。
- 「全部戻してからPR」ではなく、「上流がレビューしやすい最小差分PR」を複数に分解する。

## 2. 結論
- `PR専用ブランチを切って、名前や設定やパスを全部オリジナルへ戻してからPRする` 方針は非推奨。
- 推奨方針は、`上流追従ブランチ` を起点にして、`受け入れられる機能差分だけを小さく積む` こと。
- 命名やパスの戻しは、最初の一括リネームではなく、`PR対象機能に必要な範囲だけ` 行う。

## 3. 現状整理

### 3.1 フォーク固有差分
- リポジトリ名、ソリューション名、csproj名が `_fork` 化されている。
- `README.md` がフォーク向けの説明と温度感に寄っている。
- `IndigoMovieManager_fork.csproj` で `Title` / `AssemblyName` / `Product` / `Company` が `_fork` 前提になっている。
- `%LOCALAPPDATA%` や `%TEMP%` 配下に `IndigoMovieManager_fork` 固定パスが複数ある。
- テストプロジェクト名、`InternalsVisibleTo`、ログ出力先、ベンチ/手動確認スクリプトも `_fork` 前提になっている。

### 3.2 上流との差
- 上流は `IndigoMovieManager.sln` / `IndigoMovieManager.csproj` の単一アプリ構成が基本。
- フォーク側は `src/` 配下に Worker、Queue、Engine、UsnMft などの追加プロジェクトがある。
- 上流は `AnyCPU;x64` を持つが、フォーク側は `x64` 前提へ寄っている。
- フォーク側は高速化、Worker分離、UsnMft、Everything連携など、実装価値は高いが一度に出すとレビュー負荷が高い差分を多く含む。

## 4. 基本原則

### 4.1 一括リネームを先にやらない
- 名前を戻すだけの大差分は、レビュー上はノイズになりやすい。
- 実装改善の本質とリネーム差分が混ざると、フォーク元は安全性判断をしにくい。

### 4.2 フォーク事情と上流価値を分ける
- `フォークで共存運用するための差分` と、`上流でも価値がある改善` は分ける。
- 共存運用用の差分は、上流へそのまま送らない。
- 上流価値がある改善は、フォーク名やローカル運用事情を剥がしてから送る。

### 4.3 PRは機能単位で分割する
- 1PR 1論点を徹底する。
- 少なくとも `命名整合` `インデックス改善` `サムネ改善` `アーキテクチャ分離` は別レーンで考える。

## 5. 推奨ブランチ戦略

### 5.1 ベースブランチ
- フォーク元の最新 `master` または既定ブランチを取り込んだブランチを作る。
- 推奨名: `codex/pr/upstream-sync-20260308`
- ここでは余計な機能追加をしない。上流との差分把握専用にする。

### 5.2 PR準備ブランチ
- ベースブランチから、PRテーマごとに枝を切る。
- 例:
  - `codex/pr/readme-and-branding-cleanup`
  - `codex/pr/fileindex-provider-improvement`
  - `codex/pr/thumbnail-reliability`
  - `codex/pr/worker-separation-rfc`

### 5.3 フォーク継続ブランチ
- 既存フォークを使い続ける本流ブランチは別に維持する。
- 上流提出のために、普段使いのフォーク環境を壊さない。

## 6. 先に戻すもの / 後で戻すもの / 戻さないもの

### 6.1 先に戻すもの
- PR本文やREADMEに出る `_fork` の自己主張。
- `Title` `Product` `Company` など、上流配布物として不自然なメタ情報。
- `IndigoMovieManager_fork` 固定のログ/一時フォルダ名。
- PR対象機能に無関係なフォーク専用ドキュメント参照。

### 6.2 後で戻すもの
- `.sln` / `.csproj` / テストプロジェクト名の物理リネーム。
- `InternalsVisibleTo`、テスト名前空間、ベンチスクリプト名。
- `launchSettings.json` など、開発者向け運用名。

### 6.3 戻さないもの
- 上流でも意味がある実装改善。
- 分離したプロジェクト構造そのもの。ただし、受け入れ可否は別途説明が必要。
- バグ修正や性能改善のために追加したテスト。

## 7. PRレーンの優先順位

### レーンA: 表層整合PR
- 目的: 上流レビューの心理的障壁を下げる。
- 内容:
  - READMEを上流向けの落ち着いた説明へ寄せる。
  - フォーク専用の熱量高い記述や運用事情を外す。
  - 配布名や出力先名を必要最小限だけ上流名へ寄せる。
- 備考:
  - これは単独でも価値がある。
  - ただし、巨大なファイルリネームまでは入れない。

### レーンB: 小さな機能改善PR
- 目的: 上流が「そのまま取り込みやすい改善」を先に通す。
- 候補:
  - 既存構造を壊さないバグ修正
  - 小さな性能改善
  - テスト追加
  - ログ改善
- 備考:
  - ここで信用を作るのが重要。

### レーンC: 中規模改善PR
- 目的: FileIndex、Thumbnailの改善を段階導入する。
- 条件:
  - 上流構造へ過剰な依存を持ち込まない。
  - 新規プロジェクト追加が必須なら、先に提案文書を出す。

### レーンD: 大規模構造変更PR
- 目的: Worker分離、Queue分離、UsnMft同梱など。
- 条件:
  - いきなりコードPRではなく、先に Issue / Discussion / 設計提案で合意を取る。
- 理由:
  - 上流は現状単一アプリ構成であり、いきなり複数プロジェクト化すると受け入れコストが高い。

## 8. 具体的な実施手順

### Phase 0: 受け皿確認
1. フォーク元の最新ブランチを確認する。
2. フォーク元に `Issues` / `Discussions` / PR運用ルールがあるか確認する。
3. 現在のフォーク差分を、`提出候補` と `フォーク専用` に分類する。

### Phase 1: 上流追従ブランチを作る
1. フォーク元追従のベースブランチを作る。
2. 既存フォーク本流は触らない。
3. まずは差分一覧だけ作り、コードをすぐには戻し始めない。

### Phase 2: 表層差分を削る
1. `README.md` のフォーク専用表現を整理する。
2. アプリ名・製品名・ログ保存先など、上流で違和感が強い箇所を洗う。
3. 物理ファイル名の変更は後回しにする。

### Phase 3: 小PRへ切り出す
1. バグ修正だけを抜く。
2. テスト追加だけを抜く。
3. 性能改善だけを抜く。
4. 大規模構造変更は別枠へ退避する。

### Phase 4: 大物は提案先行
1. Worker分離
2. Queue分離
3. UsnMft内包
4. Everything依存の扱い見直し

これらは、先に「なぜ必要か」「単一プロジェクトのままでは何が限界か」を文書化してから出す。

## 9. 命名・設定・パス見直しの設計方針

### 9.1 直書きを減らす
- `IndigoMovieManager_fork` の固定文字列は定数化または設定化する。
- アプリ名、ログルート、QueueDBルート、Tempルートを分離する。

### 9.2 上流既定値を持たせる
- 既定値は `IndigoMovieManager` に寄せる。
- フォーク運用が必要なら、ローカル設定でだけ上書きできるようにする。

### 9.3 互換性を壊さない
- WhiteBrowser DB互換は維持する。
- DBファイル形式そのものを変える変更は、PR初手で入れない。

## 10. 今回の推奨判断
- ブランチは切るべき。
- ただし `全部オリジナル名へ戻すためのブランチ` ではなく、`上流へ出す差分だけを載せるブランチ` にするべき。
- 名前や設定やパスは、PR対象の本質を邪魔する部分から順に戻すべき。
- 先に出すべきは `小さく安全で説明しやすいPR`。
- 後回しにすべきは `大規模リネーム` と `大規模アーキテクチャ変更の一括投入`。

## 11. 最初の実行候補
1. `README` とアプリメタ情報の整合だけを扱うPR準備ブランチを作る。
2. `_fork` 固定の保存先・Temp先を定数化し、上流既定値へ寄せる小PRを作る。
3. その後に、バグ修正やテスト追加を機能単位で上げる。
4. Worker分離やQueue分離は、説明文書を先に出して反応を見る。

## 12. 補足
- 現在のローカルGit操作は `safe.directory` 未設定のため、そのままでは `git status` や `git remote -v` が失敗する可能性がある。
- 実運用へ入る前に、Gitの安全ディレクトリ設定を整理すること。
