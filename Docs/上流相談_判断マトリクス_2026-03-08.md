# 上流相談 判断マトリクス（2026-03-08）

## 1. 目的
大規模刷新について、上流の返答に応じて次の行動を迷わないようにする。

## 2. 判断軸
- 性能改善を本流で取り込みたいか
- 構成変更の大きさを許容できるか
- 継続保守する覚悟があるか
- 既存利用者への影響を段階管理できるか

## 3. 選択肢比較

| 選択肢 | 意味 | メリット | デメリット | 次アクション |
|---|---|---|---|---|
| A. 本流採用 | 上流が刷新方向を受け入れる | 分裂せず進められる。高速化成果を本流へ戻せる | レビュー負荷が高い。統合計画が必要 | Draft PR または統合ブランチを作る |
| B. 実験線維持 | 上流公認だが本流未統合 | 関係を保ったまま検証継続できる | 二重保守になりやすい | 実験ブランチ名と評価条件を決める |
| C. 不採用 | 上流は方向性を採らない | 判断が早い。無駄な整合作業を止められる | 本流へ戻す道は細る | フォーク本線化の方針を決める |

## 4. 返答別の推奨行動

### 4.1 A. 本流採用
- やること:
  - Draft PR か専用統合ブランチを作る
  - 命名整合は最後に回す
  - 互換性一覧とロールバック案を付ける
- やらないこと:
  - 最初に大規模リネームをする
  - 実装意図が見えなくなる無意味な分割をする

### 4.2 B. 実験線維持
- やること:
  - 上流と合意した実験線名を決める
  - 比較計測項目を固定する
  - どこまで実装が成熟したら再審査するか決める
- やらないこと:
  - 本流へ入る前提で命名だけ上流へ寄せる
  - フォーク側の改善速度を落とす

### 4.3 C. 不採用
- やること:
  - フォークを独立路線として扱う
  - 命名や設定はフォーク都合で最適化する
  - リリース方針と保守範囲を定義する
- やらないこと:
  - 受け入れられない前提の大型PR準備を続ける
  - 上流互換の見せかけ整備に時間を使う

## 5. 推奨判定基準
以下に1つでも強く当てはまるなら、最初から `小分けPR前提` を捨ててよい。
- 高速化が関数単位ではなく構成変更で成立している
- Queue、Worker、外部依存、監視方式が連動している
- 一部だけ戻すと性能か整合性が崩れる
- 上流が保守できるかどうかが最大論点である

この案件は、ほぼ上記に当てはまる。
したがって、先に問うべきは `レビュー可能か` ではなく `この路線を採るか` である。

## 6. 実務上の結論
- まず Issue / Discussion を出す
- 返答が前向きなら大型統合準備へ進む
- 返答が鈍い、または消極的ならフォーク本線化へ進む
- その前段で大規模リネームはしない
