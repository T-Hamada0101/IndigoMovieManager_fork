# Goal UI分離とスムーズ表示アーキテクチャ 2026-05-27

最終更新日: 2026-05-27

変更概要:
- UI詰まりを根本から減らすための「目指す形」を、実装計画より上位の設計ゴールとして明文化した。
- 既存コードを捨てて作り直すのではなく、既存コードを仕様の正本として参照しながら、UIの背後に新しい処理骨格を通す方針を固定した。
- Chrome系の滑らかさに近い考え方として、入力・表示・重い計算・画像供給・永続化を分ける判断基準を整理した。

## 1. この文書の目的

この文書は、IndigoMovieManager のUI詰まりを根本から減らすために、最終的に目指すアーキテクチャの形を固定するためのゴール文書である。

ここで扱うのは個別の高速化タスクではなく、今後の改修が向かうべき設計上の北極星である。

## 2. 結論

UI分離は必要である。

ただし最初に行うべき分離は、画面だけを別プロセスへ出すことではない。まず、UIスレッドを「正本処理の場」から降ろす。

目指す形は次である。

- UIスレッドは、入力、選択、スクロール、表示差分の適用だけを担当する。
- DB読込、DB保存、検索、ソート、watcher、thumbnail、metadata、skin catalog、ファイル存在確認はUIスレッドで実行しない。
- UIへ戻す結果は、全件再読込ではなく、revision付きの小さな差分にする。
- 古い検索、古いsort、古いwatch反映、古い画像要求は、後着キャンセルまたはrevision guardで捨てる。
- 見えている範囲を最優先し、見えていない範囲の画像decodeや補助UI更新は後回しにする。

## 3. なぜ今の構造は詰まるか

現在の詰まりは、単に処理が遅いからではない。

本質は、UIスレッドに次の責務が集まりすぎていることである。

- 一覧の正本管理
- 検索とソートの反映
- `Refresh()` / `Items.Refresh()` / `FilterAndSort(..., true)` による全面再評価
- DBやファイルシステムへの同期アクセス
- watcher由来の変更反映
- thumbnailと画像パス解決
- skin refreshとcatalog確認
- Player操作時の存在確認や保存処理

この状態では、どれか1つを軽くしても、別の同期処理がすぐUIスレッドを掴む。したがって「速い関数に直す」だけではなく、「UIスレッドが重い仕事を持たない構造」に変える必要がある。

## 4. 設計ゴール

### 4.1 UI Shell

UI Shell は、ユーザー入力と描画の入口である。

担当すること:
- 入力、選択、スクロール、タブ切り替えを受ける。
- 現在表示している ReadModel snapshot を描画する。
- Core から届いた diff を小さく適用する。
- 表示中の範囲やユーザー操作の優先度を Scheduler へ伝える。

担当しないこと:
- DBの同期読込や同期保存。
- ファイル存在確認。
- 大件数の検索、ソート、filter。
- thumbnail探索や画像decodeの正本判断。
- watcherやskinの重い状態確定。

### 4.2 Application Core

Application Core は、アプリの正本状態を扱う。

担当すること:
- `.wb` DBを変更せず、WhiteBrowser互換の意味論を守る。
- DB、watcher、thumbnail、skin、Player操作から来る要求を正規化する。
- UIへ返す結果を revision 付きの snapshot / diff にする。
- 処理が古くなった場合、UIへ戻す前に捨てる。

Core は UI を直接触らない。UIへ渡すのは「何をどう変えるか」という結果だけである。

### 4.3 ReadModel Store

ReadModel Store は、UI表示用の読み取りモデルを管理する。

目的:
- `MovieRecords` の正本処理と、UI表示用の状態を分ける。
- UIへ全件再代入せず、IDベースの追加、削除、更新、移動として渡す。
- 検索条件、sort条件、DB revision、source revision を明示して、古い結果を捨てられるようにする。

原則:
- 全面再評価は fallback として扱う。
- 通常経路は diff-first にする。
- sort key変更、query条件変更、大量変更、DB切替だけを全面再評価候補とする。

### 4.4 Scheduler

Scheduler は、どの仕事を先に通すかを決める。

優先順位:
1. 入力、スクロール、選択、タブ切り替え
2. Playerの明示操作
3. 現在表示中の一覧差分
4. visible範囲の画像供給
5. 検索、sortの確定結果
6. watcherの小差分反映
7. thumbnail生成、rescue、prewarm、catalog再確認

重要なのは、背後処理を速くすることだけではない。ユーザー要求が来た時に、背後処理が一歩引くことである。

### 4.5 Image Pipeline

Image Pipeline は、サムネイルと画像表示を visible-first にする。

担当すること:
- 表示中の範囲を最優先する。
- off-screen decode を入力やスクロールより優先しない。
- ファイル存在確認、stamp取得、画像decodeをUIスレッドで行わない。
- cache miss 時も、UIを待たせず placeholder で先に返す。
- 成功、失敗、未生成、ERROR marker を revision 付きで反映する。

### 4.6 Persistence Pipeline

Persistence Pipeline は、保存処理をUI操作から分離する。

対象:
- 設定保存
- 再生回数やscore更新
- tag変更
- bookmark追加、削除
- skin profile保存
- DB上のパス更新

UIは先に表示値を反映し、保存は背景で直列化する。保存失敗時だけ、ログと必要最小限のUI通知へ戻す。

## 5. Chrome系の滑らかさから借りる考え方

Chromeは、すべての処理を1本のUIスレッドで終わらせてから描画しているわけではない。

参考にする考え方は次である。

- 入力を重いページ処理より優先する。
- 描画や合成を、重いメイン処理から分ける。
- 古いフレームを表示しながら、次のフレームを準備する。
- 画像decodeやrasterを並列化し、表示に必要な部分から進める。
- renderer、browser、GPU、compositor のように、責務を分けて障害と遅延を閉じ込める。

IndigoMovieManager では、これを次のように読み替える。

- browser process 相当: アプリ全体の状態管理と要求調停。
- renderer process 相当: 検索、ソート、ReadModel生成。
- compositor 相当: UI Shell と差分適用。
- GPU / raster 相当: thumbnail と画像供給。

すぐに別プロセスへ分ける必要はない。まず同一プロセス内で、この責務分離を作る。

## 6. 書き直しか、既存修正か

丸ごと書き直しはしない。

既存コードには、WhiteBrowser互換、`.wb` DBの癖、skin API、サムネイル、watcher、Player、失敗動画対応など、仕様書に書き切れない現実が埋まっている。

したがって方針は次で固定する。

- 既存コードは仕様の正本として参照する。
- 新しい Core / ReadModel / Scheduler / ImagePipeline を横に作る。
- 既存UIから少しずつ新しい背骨へ接続を差し替える。
- `Refresh()` / `Items.Refresh()` / `FilterAndSort(..., true)` を通常経路から外していく。
- UI、DB、skin、Playerの外部契約は急に壊さない。

これは、稼働中のアプリを止めずに内部構造を置き換える Strangler 型の移行である。

## 7. 最初に新設計へ寄せる場所

最初に狙うべき場所は、画面全体ではなく一覧更新の心臓部である。

優先順:
1. `MovieRecords` 正本とUI表示用 ReadModel を分ける。
2. 検索とソートを背景で revision 付き実行にする。
3. UIへは全件再代入ではなく diff を返す。
4. 画像は visible-first pipeline に寄せる。
5. watcher / thumbnail / skin は直接UIを触らず、Schedulerへ要求を投げる。

ここが通れば、以後の Player、skin、thumbnail、watcher の改善が同じ背骨に乗る。

## 8. 禁止線

次の変更は、原則として採らない。

- `.wb` DBの互換性を壊す。
- UIテンポ改善を理由に観測ログを削る。
- `MainWindow` に新しい正本責務を戻す。
- 背景化した結果を、最後にUI上の全面 `Refresh()` で相殺する。
- 別プロセス化だけでUI詰まりが解決すると見なす。
- skin、watcher、thumbnail、Playerを個別最適化し、一覧ReadModelの正本設計と衝突させる。

## 9. 完了条件

このゴールに近づいたと判断できる条件は次である。

- 入力、スクロール、選択、タブ切り替えが、watcher / thumbnail / skin / DB保存に押し負けない。
- watchの1件追加やrenameで、通常経路として全面 reload を踏まない。
- 大件数検索やsortの古い処理が、後着入力後もCPUを食い続けない。
- UIへ戻る変更が、revision付き diff として説明できる。
- visible範囲外の画像decodeが、visible範囲や入力を押しのけない。
- `debug-runtime.log` で、詰まりの支配要因を search / sort / watch / image / skin / persist に分けて説明できる。
