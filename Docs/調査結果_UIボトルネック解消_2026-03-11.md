# 調査結果（UIボトルネック解消）

最終更新日: 2026-03-11

## 1. 調査目的

- 一覧表示まわりの「重さ」の主因がどこかを切り分ける。
- `git` や監視改善とは別に、通常操作時のUI体感を改善する優先ポイントを整理する。
- 次の実装修正で何から着手すべきかを、コード根拠付きで残す。

## 2. 結論

- UIボトルネックの本命は「仮想化不足」ではない。
- 本命は `MainWindow.xaml.cs:3062` `FilterAndSort` を中心にした以下の全件更新パターンである。
  - UIスレッド上での同期 `SELECT *`
  - `MovieRecs.Clear()` からの全件再構築
  - `ObservableCollection` への全件コピー
  - 5ビューへの `ItemsSource` 再代入
  - 各ビューの `Items.Refresh()` 強制実行
  - 先頭選択の自動変更に伴う詳細パネル更新と副作用
- つまり、今効くのは「描画部品を増やすこと」より「全件再バインドをやめること」である。

## 2.1 2026-03-11 の初手対応

- 初手として、`ItemsSource` 再代入と `Items.Refresh()` は「選択中タブだけ」に絞る修正を実施した。
- さらに、自動 `SelectFirstItem()` は廃止し、ユーザーがクリックするまでは無選択を許容する方針へ寄せた。
- 非表示タブは、タブ切替時に `filterList` を反映する既存経路で追従させる。
- これで全件再構築自体は残るが、非表示4ビューぶんの無駄な再描画は先に削減できる。

## 2.2 2026-03-11 の次対応

- `MainWindow.xaml` の `FilteredMovieRecs` バインドを、コード側でも正として扱うよう揃えた。
- `MainWindow.xaml.cs` の `filterList = new ObservableCollection<MovieRecords>(MainVM.MovieRecs)` を撤去した。
- `FilterAndSort` は `MainWindowViewModel.FilterMovies(...)` と `SortMovies(...)` を使って `FilteredMovieRecs` を作り直す形へ変更した。
- `SortData` も同じく `FilteredMovieRecs` の並び替えに限定し、再検索や `ItemsSource` 再代入を避けるようにした。
- `Refresh()` から `Items.Refresh()` を外し、詳細ペイン同期だけに責務を絞った。
- `Watcher` 側の表示中一覧スナップショットも `FilteredMovieRecs` 参照へ揃えた。
- `FilterAndSort` の `SELECT * FROM movie ...` は `Task.Run(...)` で UI スレッド外へ逃がし、読込完了時だけ UI へ戻す形にした。
- 非同期読込の途中で別条件の検索や再読込が入っても、古い結果で画面を巻き戻さないよう request revision を入れた。

これで「全件コピーした別コレクションをさらに各タブへ再代入する」経路は外れた。
また、DB再取得そのものも UI 待ちから外れた。
まだ `MovieRecs.Clear()` からの全件再構築自体は残るため、次は `MovieRecs` 差し替えの粒度削減が候補になる。

## 3. 現状整理

### 3.1 仮想化は既にかなり入っている

- `MainWindow.xaml:14` で `VirtualizingWrapPanel` を使用している。
- `MainWindow.xaml:293` `VirtualizingPanel.IsVirtualizing="True"`
- `MainWindow.xaml:337` `VirtualizingPanel.VirtualizationMode="Recycling"`
- `MainWindow.xaml:562` `EnableRowVirtualization="True"`
- `MainWindow.xaml:866` でも `DataGrid` 側の仮想化を有効化している。

このため、現状を「仮想化していないから重い」とみなすのはズレている。
仮想化より強く効いてしまっているのが、一覧全体の再構築と再描画である。

### 3.2 一部タブは既にUI負荷対策が入っている

- 進捗タブは `MainWindow.xaml.cs:1257` `RequestThumbnailProgressSnapshotRefresh()` で `DispatcherPriority.Background` に寄せて更新要求を集約している。
- 進捗表示のログ差分反映は `ModelViews/ThumbnailProgressViewState.cs:196` `SyncQueueLogs()` で内容同一時の更新を避けている。
- サムネ失敗タブは `Thumbnail/MainWindow.ThumbnailFailedTab.cs:67` `RequestThumbnailFailedListRefresh()` で連続イベントを1本化している。
- 同タブは `Thumbnail/MainWindow.ThumbnailFailedTab.cs:542` `CollectionViewSource.GetDefaultView(...)` を使い、`ItemsSource` の全差し替えではなく `CollectionView` で絞り込みをかけている。

つまり、重い中心は進捗タブや失敗タブではなく、メイン一覧の旧来更新経路である。

## 4. 主なボトルネック

### 4.1 `FilterAndSort` が UI スレッド上で同期DB読込を行う

- `MainWindow.xaml.cs:3070-3078`
- `movieData == null || IsGetNew` の時、`GetData(..., "SELECT * FROM movie ...")` を即実行している。
- 起動直後や再読込系操作で、この同期読込がそのままUI待ち時間になる。

### 4.2 DB結果を毎回 `MainVM.MovieRecs` へ全件再構築している

- `MainWindow.xaml.cs:3089`
- `MainWindow.xaml.cs:3624-3635`
- `SetRecordsToSource()` で `MainVM.MovieRecs.Clear()` 後、全行を `DataRowToViewData(row)` で詰め直している。

件数が増えるほど、GC・通知・バインディング再評価がまとめて重くなる。

### 4.3 表示用コレクションを毎回フルコピーしている

- `MainWindow.xaml.cs:3097`
- `filterList = new ObservableCollection<MovieRecords>(MainVM.MovieRecs);`

元データを再構築した直後に、表示用としてさらに全件コピーしており、一覧件数が大きいほど二重コストになる。

### 4.4 フィルタ後に 5ビューへ `ItemsSource` を再代入している

- `MainWindow.xaml.cs:3234-3238`
- `SmallList` / `BigList` / `GridList` / `ListDataGrid` / `BigList10` のすべてに再代入している。

仮想化が有効でも、`ItemsSource` 自体を丸ごと差し替えると、ビュー側はかなり広い範囲を再評価する。

### 4.5 `Refresh()` が全ビューを強制再描画している

- `MainWindow.xaml.cs:2911-2925`
- `SmallList.Items.Refresh()`
- `BigList.Items.Refresh()`
- `GridList.Items.Refresh()`
- `ListDataGrid.Items.Refresh()`
- `BigList10.Items.Refresh()`

しかも最後に `viewExtDetail.DataContext = mv;` を再設定しており、一覧更新に詳細更新までぶら下がっている。

### 4.6 ソートだけでも全ビュー再代入 + 全ビュー `Refresh()` を踏む

- `MainWindow.xaml.cs:3250-3335` `SetSortData`
- `MainWindow.xaml.cs:3341-3358` `SortData`

ソート変更は本来 `CollectionView.SortDescriptions` やソート済みビュー差し替えで済ませたい処理だが、現状は一覧全再適用になっている。

### 4.7 自動先頭選択が `SelectionChanged` 副作用を引き起こす

- `MainWindow.xaml.cs:2858-2905` `SelectFirstItem`
- `MainWindow.Selection.cs:20-51` `List_SelectionChanged`

`SelectFirstItem()` は各タブで `SelectedIndex = 0` を入れる。
その結果 `SelectionChanged` が走り、以下が連動する。

- `viewExtDetail.DataContext = mv`
- `ThumbDetail` に `error` を含む場合の `TryEnqueueThumbnailJob(...)`

つまり「一覧を更新しただけ」で、詳細更新やサムネ再投入まで飛ぶ経路がある。

## 5. 二次候補

### 5.1 サムネイル画像デコード

- `Converter/NoLockImageConverter.cs:18-21` で LRU キャッシュはある。
- `Converter/NoLockImageConverter.cs:92-94` で `DecodePixelHeight` 指定にも対応している。
- `Converter/NoLockImageConverter.cs:225-231` で指定時のみ縮小デコードしている。

ただし、現時点では主因より一段弱い。
まずは一覧の全件再バインドを止め、その後も残る場合に着手するのが順序として妥当である。

## 6. 改善方針

### 6.1 最優先: メイン一覧を「1本の表示コレクション + View」で扱う

- `ModelViews/MainWindowViewModel.cs:158` `ReplaceFilteredMovieRecs(...)`
- `ModelViews/MainWindowViewModel.cs:170` `FilterMovies(...)`
- `ModelViews/MainWindowViewModel.cs:285` `SortMovies(...)`

すでに ViewModel 側に、移行先として使いやすい部品がある。
`FilterAndSort` の責務を MainWindow 側に溜めず、以下へ寄せるのがよい。

- 元データ: `MovieRecs`
- 表示用: `FilteredMovieRecs`
- 絞り込み: `FilterMovies(...)`
- 並び替え: `SortMovies(...)`
- 反映: `ReplaceFilteredMovieRecs(...)`

重要なのは、各タブで `ItemsSource` を差し替え続けない構造へ変えること。

### 6.2 次点: DB再取得を UI スレッド外へ逃がす

- `SELECT * FROM movie ...` はバックグラウンド取得し、UI側は結果適用だけに絞る。
- 少なくとも `IsGetNew = true` 系の再読込で、UIフリーズを減らせる。

### 6.3 次点: `CollectionView` パターンへ寄せる

失敗タブが採っている以下の形が比較対象として良い。

- dirty フラグで再読込要求を集約する
- `Dispatcher.BeginInvoke(...)` で連続イベントを1本化する
- `CollectionViewSource.GetDefaultView(...)` に `Filter` を設定する
- 必要時だけ `view.Refresh()` する

メイン一覧もこの方向へ寄せれば、検索文字変更やソート変更の UI 負荷をかなり抑えやすい。

### 6.4 自動選択を必要時だけに限定する

- 再検索やソート変更のたびに `SelectedIndex = 0` を入れない。
- 選択中アイテムがまだ結果内に存在するなら維持する。
- 初回表示、または選択対象消失時だけ先頭選択する。

これだけでも `SelectionChanged` 由来の余計な詳細更新と再投入をかなり減らせる。

### 6.5 詳細パネル更新を一覧更新から分離する

- `Refresh()` 内で毎回 `viewExtDetail.DataContext = mv` を触らない。
- 選択変更時のみ詳細を更新する。
- `ThumbDetail` エラー時の再投入も、必要ならデバウンスや明示操作へ寄せる。

## 7. 優先順位

1. `FilterAndSort` の全件再構築・全タブ再代入・全ビュー `Refresh()` をやめる。
2. DB再読込を UI スレッド外へ逃がす。
3. `SelectFirstItem()` と `SelectionChanged` の副作用を絞る。
4. 必要ならサムネ画像の `DecodePixelHeight` 最適化を追加する。

## 8. 実装案の叩き台

### 案A: 低リスク寄り

- XAML 各タブの `ItemsSource` を `MainVM.FilteredMovieRecs` へ固定する。
- `FilterAndSort` は `MainVM.MovieRecs` を元に `FilterMovies` -> `SortMovies` -> `ReplaceFilteredMovieRecs` へ縮退する。
- `Refresh()` の全ビュー `Items.Refresh()` を撤去し、必要箇所だけ `CollectionView.Refresh()` に置き換える。

まずここまででも、一覧件数が多い時の再描画量を大きく減らせる。

### 案B: 本命

- `MovieRecs` に対する `ICollectionView` を1つ持つ。
- 検索は `view.Filter`、並び替えは `SortDescriptions` またはソート済み差し替えで扱う。
- 再読込要求は dirty / revision / debounce でまとめる。

保守性まで含めるとこちらが本命だが、既存コードへの入り方は案Aより広い。

## 9. 計測観点

改善前後は、少なくとも以下を同条件で比較したい。

- 起動直後の初回一覧表示時間
- 検索キーワード変更時の応答時間
- ソート変更時の応答時間
- 監視追加後の一覧反映時間
- タブ切替時の応答時間

ログに残すと有効な区間は以下。

- `FilterAndSort` 全体時間
- DB取得時間
- `SetRecordsToSource` 時間
- 表示用コレクション生成時間
- `Refresh()` 時間
- `SelectionChanged` 発火回数

## 10. まとめ

- すでに仮想化や一部タブのUI負荷対策は入っている。
- それでも重い理由は、メイン一覧が「全件再読込 -> 全件再構築 -> 全件再バインド -> 全ビュー強制更新」の古い流れを持っているためである。
- 次の改修は、まず `FilterAndSort` 周辺を `FilteredMovieRecs` / `CollectionView` ベースへ寄せるのが最も費用対効果が高い。
