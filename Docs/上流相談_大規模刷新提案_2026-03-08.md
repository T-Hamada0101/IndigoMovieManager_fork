# 上流相談: 大規模刷新提案（2026-03-08）

## 1. これは何か
この文書は、`IndigoMovieManager_fork` で進めてきた大規模刷新を、フォーク元が受け入れる意思があるか事前相談するための提案書である。

結論から言う。
これは小分けPRで自然吸収する類ではない。
現行の単一アプリ構成を保ったままの局所改善ではなく、監視、サムネイル、キュー、外部依存の扱いまで跨ぐ刷新である。

そのため、先に確認したい論点は `実装の細部` ではなく、次の3点である。
- この刷新方向そのものを上流が採る意思があるか
- 本流へ即時統合するのか、実験ブランチで持つのか
- 不採用なら、フォークを独立路線として維持するか

## 2. 提案したい刷新の中身

### 2.1 監視とインデックス取得の刷新
- 監視フォルダの更新検知を、単純な `FileSystemWatcher` 依存から拡張した。
- `Everything`、`USN-MFT`、通常ファイルシステムの複数プロバイダを使い分ける構成を導入した。
- これにより、全件DB読込ベースの差分判定を減らし、ファイル実体ベースの差分抽出へ寄せた。

### 2.2 サムネイル生成系の刷新
- サムネイル作成を同期的なUI従属処理から分離し、キュー駆動へ寄せた。
- エンジンは `autogen` / `ffmediatoolkit` / `ffmpeg1pass` / `opencv` を切替可能にした。
- 絵文字パスや高負荷条件を踏まえたルーティングを導入した。
- 失敗時の救済経路、エラーマーカー、再試行制御を強化した。

### 2.3 ワーカー分離と責務整理
- サムネイル処理を本体プロセスから切り離しやすい構成へ進めた。
- Queue、Engine、WorkerCore、ProgressViewer などを分離し、重い処理をUI本体から外す方向で整理した。
- まだ最終形ではないが、少なくとも「重い処理を本体に抱え込む前提」からは脱している。

### 2.4 互換性を残した改善
- WhiteBrowser DB互換は維持する方針で進めている。
- 既存データ資産を壊すための刷新ではなく、処理方式を刷新するための改造である。

## 3. なぜ単純な小分けPRにしにくいか
- 監視、DB登録、サムネイル投入、サムネイル生成、UI反映が連動している。
- 片側だけ出すと、設計意図が崩れるか、上流側で再統合作業が発生する。
- `高速化` が単一関数の改善ではなく、構成変更の積み上げで成立している。
- 名前だけ戻しても、本質は `大規模な実装方針の採否` の問題であり、レビュー負荷は下がらない。

## 4. 現時点で示せる効果

### 4.1 サムネイル生成
- `Docs/1Panel_EngineBench_2026-03-01.md` の検証では、小ファイル1パネル生成で `autogen` の平均は `110ms`。
- 同条件で `ffmediatoolkit` は `226ms`、`ffmpeg1pass` は `342ms`。
- 少なくとも小ファイル・1パネルの主力経路では、主力エンジン切替に妥当性がある。

### 4.2 ハッシュ処理
- `Thumbnail/Hash取得ベンチ結果_2026-02-25.md` の検証では、`System.IO.Hashing` ベースのCRC32は旧ロジックに対して約5倍高速で、既存ハッシュ値互換を維持した。

### 4.3 全体構成としての狙い
- UIの重い同期処理を減らす。
- 監視とサムネイル作成を、DB中心の詰まりやUI巻き込みから切り離す。
- 失敗時に無限再試行や再スキャン地獄へ入らないよう、防御線を張る。

## 5. 上流が判断すべきこと
上流にお願いしたいのは、コードレビュー以前の方針判断である。

### Option A: 本流採用を前提に進める
- 方針として受け入れる。
- 段階統合のための受け皿ブランチ、または Draft PR を用意する。
- 互換性整理や命名整合はその後に詰める。

### Option B: 実験ブランチとして保持する
- 本流へはまだ入れない。
- ただし、上流公認の実験線として枝を持つ。
- 効果検証と保守性評価を続ける。

### Option C: 本流では採らない
- 方向性が違うと判断する。
- この場合は、フォーク側で独立路線として育てる。
- 無理に上流命名へ戻したり、PRの体裁だけ整える作業はやらない。

## 6. 私見
現実的には `Option A` か `Option B` のどちらかを先に問うべきである。
`Option A` なら大型PR準備へ進む。
`Option B` なら比較運用と計測を続ける。
`Option C` なら、フォークを本線として育てる判断を早めるべきである。

中途半端に「名前だけ戻して通常PRのふりをする」のが一番悪い。
それは上流にもフォーク側にも無駄なコストを発生させる。

## 7. 提案する次アクション
1. まず Issue か Discussion で、この刷新の受け入れ可否を相談する。
2. 相談時には、次を添える。
   - 全体構成図
   - 何を置き換えたかの要約
   - ベンチの要点
   - 互換性維持点と非維持点
   - 上流に求める判断
3. 回答が前向きなら Draft PR または専用統合ブランチへ進む。
4. 消極的なら、フォーク本線化へ舵を切る。

## 8. 上流への問い
上流へは、次の形で聞くのがよい。

- 現行の単一構成を保った局所修正ではなく、監視・サムネイル・キューを含む大規模刷新である。
- 小分け吸収ではなく、刷新方向そのものの採否を相談したい。
- 本流採用、実験ブランチ保持、不採用のどれを望むか。
