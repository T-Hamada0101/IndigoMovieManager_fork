# AI向け ブランチ方針: サムネ高速感優先 2026-03-11

最終更新日: 2026-03-11

変更概要:
- 初版作成
- このブランチで AI が守るべき思想、採用基準、禁止事項を固定

## 1. この文書の目的

この文書は、AI エージェントがこのブランチで改修する時に、機能追加や設計改善よりも先に守るべき判断軸を固定するための資料である。

特にサムネイルまわりは、機能的に正しくても「速く進んでいる感覚」が失われると価値が大きく落ちる。  
そのため、このブランチでは単純な正しさや設計の綺麗さよりも、体感速度と安定運用を優先する。

## 2. このブランチの立場

- このブランチは、サムネイルの高速感を第一にした安定化ラインとして扱う。
- 他ブランチに改善が存在しても、丸ごと統合してよい枝ではない。
- 他ブランチの変更は「改善候補の倉庫」として扱い、このブランチには必要な部品だけを選別して取り込む。
- 取り込み判断は、コード量や実装の新しさではなく、このブランチの目的に合うかで決める。

## 3. 最優先の判断軸

優先順位は以下で固定する。

1. サムネイルの高速感を壊さない
2. UI の引っかかりを増やさない
3. 安定して復旧・再試行・フォールバックできる
4. 既存運用を壊さない
5. その上で改善を取り込む

ここでいう「高速感」とは、単純な平均処理時間だけではない。少なくとも以下を含む。

- 監視や投入の直後に、すぐ動き始めたと分かること
- 最初の数件が早く見えること
- 重い動画が混ざっても、全体が止まったように見えないこと
- UI 操作中にカーソルやスクロールの引っかかりを増やさないこと

## 4. 採用してよい変更

優先して採用してよいのは、次のような変更である。

- 明確なバグ修正
- テスト追加
- ログ追加、原因切り分け強化
- 設定追加。ただし既定値で高速感を悪化させないもの
- DTO、契約、型整理
- フォールバック改善
- 失敗時の安全化
- UI スレッドを塞がないための局所改善

## 5. 慎重に扱う変更

以下は価値があっても、そのまま入れてはいけない。

- キュー制御の大幅変更
- 並列数制御の思想変更
- Watcher の走査戦略変更
- UI 反映タイミング変更
- 自動救済の発火条件変更
- 常駐サービス化や IPC 実装の本格導入
- 大規模リファクタ

これらを入れる場合は、必ず「体感速度をどう守るか」を先に説明し、変更を小さく分割すること。

## 6. 原則禁止

AI は以下を原則として行ってはいけない。

- 改善が多いという理由だけで他ブランチを丸ごと merge する
- 巨大コミットを意味単位に分けずに取り込む
- 測定や確認なしで並列制御や投入条件を変える
- 正しさのために体感速度の悪化を黙って受け入れる
- 未来の理想設計を優先して、現行の速さを犠牲にする

## 7. 他ブランチからの取り込みルール

- 基本は `merge` ではなく `cherry-pick` か手動 backport を使う
- 取り込み単位はコミット単位ではなく、責務単位で見る
- 1 回の作業で混ぜるテーマは 1 つに寄せる
- 速度に効く変更と、速度に効かない変更を同時に入れない
- 迷ったら「今回は入れない」を選ぶ

### 7.1 先に取り込みやすいもの

- テスト
- DTO、契約
- ログ
- エラー分類
- フォールバック理由の明確化
- 設定値の追加

### 7.2 後で個別検証するもの

- サムネイル並列アルゴリズム
- レーン予約
- Watcher 再突合や救済条件
- UI 逐次反映ロジック
- IPC 実通信

## 8. AI が変更前に確認すること

サムネイル、Watcher、Queue に触る時は、少なくとも以下を意識すること。

- 最初の数件が早く見えるか
- 重い 1 件に全体が引きずられないか
- UI スレッドへ重い処理を戻していないか
- フォールバック時に静かに劣化しすぎていないか
- 自動救済が通常キューを邪魔しないか
- ログだけ見れば、遅くなった理由を追えるか

## 9. AI への明示指示

- このブランチでは「速さこそ正義」を雑に言い換えてはならない。守るべきは体感速度と安定運用の両立である。
- 速度改善を名目に可観測性を削ってはならない。
- 安定化を名目に、初動の速さを殺してはならない。
- 変更提案時は、必ず「高速感への影響」を1段落で説明すること。
- 他ブランチ由来の変更を提案する時は、丸ごと導入ではなく、何を捨てて何を採るかを明記すること。

## 10. 更新ルール

- このブランチの目的が変わったら、まずこの文書を更新する
- 方針に反する例外を入れる場合は、理由を文書へ追記する
- 体感速度に関する判断軸が増えたら、禁止事項より先にこの文書へ反映する
